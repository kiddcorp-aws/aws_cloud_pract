Lab 4: Working with S3 Buckets and Objects

Duration: 45 minutes

Level: Introductory

Prerequisites: Completed Labs 1-3; AWS console access with S3 permissions

Lab Objectives

  • Create an S3 bucket with specific configuration
  • Upload objects and organize them with prefixes (folders)
  • Enable versioning and observe version behavior
  • Configure a bucket policy for public read access
  • Enable static website hosting
  • Manage S3 buckets and objects using the AWS CLI
  • Configure lifecycle rules to automate storage transitions
  • Explore encryption options and storage classes
  • Generate and test presigned URLs for temporary access
  • Clean up all resources

Part A: Create an S3 Bucket

1 In the AWS Management Console, type S3 in the search bar at the top and click S3 from the results.

2 Click Create bucket and configure:

  • Bucket name: cloud-pract-lab-<your-initials>-<random> (bucket names must be globally unique)
  • Region: US East (N. Virginia) us-east-1
  • Object Ownership: ACLs disabled (recommended)
  • Block Public Access: Leave all checked for now (we will change this later)
  • Bucket Versioning: Disable (we will enable it later)
  • Default encryption: SSE-S3 (Amazon S3 managed keys)
  • Click Create bucket

Part B: Upload Objects

3 Click on your new bucket to open it. Click Upload.

4 Create a simple HTML file on your local machine called index.html:

<!DOCTYPE html>
<html>
<head><title>My S3 Website</title></head>
<body>
    <h1>Hello from Amazon S3!</h1>
    <p>This page is hosted on S3 static website hosting.</p>
</body>
</html>

You can create this file using any text editor (Notepad, TextEdit, VS Code) and save it to your Desktop.

5 Create a second file called error.html with the following content:

<!doctype html>
<html>
<head><title>Error</title></head>
<body>
    <h1>404 — Page Not Found</h1>
    <p>The requested page does not exist.</p>
</body>
</html>

Upload both index.html and error.html to the bucket.

6 Click Create folder and create a folder called images. Upload any small image file into this folder. Note how S3 uses the folder name as a prefix in the object key.

Part C: Enable Versioning

7 Go to the bucket's Properties tab. Under Bucket Versioning, click Edit and enable versioning. Save changes.

8 Modify your index.html locally (change the heading text) and upload it again to the same bucket.

9 In the bucket's Objects tab, toggle Show versions. You should see two versions of index.html, each with a unique Version ID.

Part D: Static Website Hosting

10 Go to the bucket's Properties tab. Scroll to Static website hosting and click Edit:

  • Static website hosting: Enable
  • Index document: index.html
  • Error document: error.html
  • Save changes. Note the Bucket website endpoint URL.

11 Go to the Permissions tab. Under Block public access, click Edit and uncheck all options. Save and confirm.

12 Still on the Permissions tab, scroll to Bucket policy and click Edit. Paste the following policy (replace <first-bucket> with your bucket name):

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": "*",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::<first-bucket>/*"
        }
    ]
}

13 Open the Bucket website endpoint URL in a new browser tab. You should see your index.html page served from S3.

Part E: S3 Operations with AWS CLI

14 Open CloudShell by clicking the terminal icon in the top navigation bar of the AWS Console. Wait for the shell to initialize (you will see a command prompt). Alternatively, if your EC2 instance from Lab 3 is still running, you can use EC2 Instance Connect.

15 Create a second S3 bucket using the CLI:

aws s3 mb s3://cloud-pract-cli-<your-initials>-<random>

Replace <your-initials> and <random> with your own values. Confirm the bucket was created successfully.

Note: In all CLI commands below, replace <cli-bucket> with the full name of the bucket you just created (e.g., cloud-pract-cli-jk-42857). Replace <first-bucket> with the name of the bucket you created in Part A.

16 Create a local file and upload it to your CLI bucket:

echo "uploaded via CLI" > cli-test.txt
aws s3 cp cli-test.txt s3://<cli-bucket>/

17 List the objects in your CLI bucket:

aws s3 ls s3://<cli-bucket>/

18 Copy an object between your two buckets:

aws s3 cp s3://<cli-bucket>/cli-test.txt s3://<first-bucket>/cli-copy.txt

Verify the copied object appears in your first bucket using the console or CLI.

19 Sync a local directory to S3. First, create a directory with a few files:

mkdir testdir && echo "file1" > testdir/a.txt && echo "file2" > testdir/b.txt
aws s3 sync testdir/ s3://<cli-bucket>/testdir/

20 List objects with a prefix to see only the synced directory contents:

aws s3 ls s3://<cli-bucket>/testdir/

Part F: Configure a Lifecycle Rule

21 Go back to your first S3 bucket in the AWS console. Navigate to the Management tab and click Create lifecycle rule.

22 Configure the lifecycle rule:

  • Lifecycle rule name: archive-old-objects
  • Choose a rule scope: Apply to all objects in the bucket
  • Check the acknowledgement box

23 Under Lifecycle rule actions, check the following:

  • Transition current versions of objects between storage classes
  • Expire current versions of objects

24 Configure the transitions for current versions:

  • Add transition: Move to Standard-IA after 30 days
  • Add transition: Move to Glacier Flexible Retrieval after 90 days

25 Configure expiration: Set current versions to expire after 365 days. Click Create rule.

26 On the Management tab, click on your lifecycle rule name (archive-old-objects). Verify that the rule shows transitions to Standard-IA at 30 days and Glacier at 90 days, with expiration at 365 days. This automates cost optimization without manual intervention.

Part G: Explore S3 Encryption and Storage Classes

27 In your first bucket, click on one of your existing objects (e.g., index.html). Scroll to Server-side encryption settings. Note that it shows SSE-S3 (Amazon S3 managed keys) — this is the default encryption.

28 Upload a new object (create a small text file called kms-test.txt). During upload, expand the Properties section. Under Server-side encryption, select Specify an encryption key, choose AWS Key Management Service key (SSE-KMS), and select the default aws/s3 key. Complete the upload.

29 Compare the encryption settings between your original object and the KMS-encrypted object by clicking on each and checking their properties. Note the different encryption types.

30 Upload another object (create ia-test.txt). During upload, expand Properties and change the Storage class to S3 Standard-IA. Complete the upload.

31 Click on ia-test.txt and view its properties. Confirm the Storage class shows Standard-IA. Compare this with your other objects, which should show Standard.

Part H: Generate a Presigned URL

32 Return to CloudShell. Generate a presigned URL for one of your objects that expires in 300 seconds (5 minutes):

aws s3 presign s3://<first-bucket>/index.html --expires-in 300

33 Copy the generated URL and open it in a new browser tab. Try opening it in an incognito/private window as well. The page should load successfully even without AWS credentials, because the presigned URL contains temporary authorization.

34 To see expiration in action, generate a URL with a very short lifetime:

aws s3 presign s3://<first-bucket>/index.html --expires-in 60

Open it immediately (should work), then wait 60 seconds and try the same URL again — it should return an Access Denied error.

35 Compare the presigned URL structure with a normal S3 URL. Notice the query parameters that include the signature and expiration. Presigned URLs provide temporary, time-limited access to private S3 objects without making them public or sharing AWS credentials.

Part I: Clean Up

Important: You must empty each bucket before you can delete it. For versioned buckets, you must delete all object versions.

36 Go to your first bucket in the console. Toggle Show versions, select all objects and versions, and click Delete. Confirm the deletion.

37 Once the bucket is empty, go back to the S3 bucket list, select your first bucket, and click Delete. Type the bucket name to confirm.

38 Delete the CLI bucket using CloudShell. The --force flag empties the bucket and deletes it in one command:

aws s3 rb s3://<cli-bucket> --force

Lab Complete

You have created S3 buckets via console and CLI, uploaded objects, enabled versioning, configured static website hosting with a bucket policy, managed objects with the AWS CLI, configured lifecycle rules, explored encryption and storage classes, generated presigned URLs, and cleaned up all resources.

Discussion Questions

  1. Why are S3 bucket names globally unique across all AWS accounts?
  2. How does versioning protect against accidental deletion or overwrite?
  3. What are the security implications of making a bucket publicly accessible?
  4. How could you use S3 lifecycle policies to reduce storage costs over time?
  5. How would you design a lifecycle policy for a company that needs to keep compliance data for 7 years but rarely accesses it after 90 days?
  6. In what scenarios would presigned URLs be preferable to making objects public or using IAM-based access?
