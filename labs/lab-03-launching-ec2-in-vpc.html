Lab 3: Launching an EC2 Instance in a VPC

Duration: 45 minutes

Level: Introductory

Prerequisites: Completed Labs 1-2; AWS console access with EC2 and VPC permissions

Lab Objectives

  • Explore the default VPC and its components
  • Review subnets, route tables, and internet gateways
  • Launch an EC2 instance with a security group
  • Connect to the instance using EC2 Instance Connect
  • Query instance metadata using both IMDSv1 and IMDSv2
  • Modify security group rules and observe the effects on connectivity
  • Create, attach, format, and mount an EBS volume
  • Create an EBS snapshot for backup
  • Terminate the instance and clean up all resources
Important: Use the us-east-1 region for this lab. Ensure you terminate all resources at the end to avoid charges.

Part A: Explore the Default VPC

1 Navigate to VPC in the AWS Management Console. Ensure you are in us-east-1.

2 In the left navigation, click Your VPCs. You should see a default VPC. Note its:

  • VPC ID
  • IPv4 CIDR block (typically 172.31.0.0/16)
  • State (Available)

3 Click Subnets in the left navigation. Note that the default VPC has one subnet per Availability Zone. Each is a public subnet by default.

4 Click Internet gateways. Note that the default VPC has an internet gateway attached, which is what makes the subnets public.

5 Click Route tables. Select the main route table for the default VPC. In the Routes tab, note:

  • A local route for the VPC CIDR (172.31.0.0/16)
  • A route to 0.0.0.0/0 pointing to the internet gateway

Part B: Launch an EC2 Instance

6 Navigate to EC2 and click Launch instance.

7 Configure the instance:

  • Name: lab-web-server
  • AMI: Amazon Linux 2023 (Free tier eligible)
  • Instance type: t2.micro (Free tier eligible)
  • Key pair: Select Proceed without a key pair (we will use EC2 Instance Connect)

8 Under Network settings, click Edit:

  • VPC: Select the default VPC
  • Subnet: Select any available subnet
  • Auto-assign public IP: Enable
  • Security group: Create a new security group named lab-web-sg
  • Add an inbound rule: Type: HTTP, Source: 0.0.0.0/0
  • Keep the default SSH rule (port 22) from anywhere

9 Expand Advanced details and paste the following in the User data field:

#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello from $(hostname -f)</h1><p>AWS Cloud Practitioner Lab 3</p>" > /var/www/html/index.html

10 Leave all other settings as default and click Launch instance.

Part C: Verify and Connect

11 Wait for the instance state to change to Running and the status checks to show 2/2 checks passed (this may take 1-2 minutes).

12 Select the instance and note its Public IPv4 address. Open a new browser tab and navigate to http://<public-ip>. You should see the "Hello from..." web page.

13 Back in the EC2 console, select the instance, click Connect, and use EC2 Instance Connect to open a browser-based terminal. Run:

curl http://169.254.169.254/latest/meta-data/instance-type
curl http://169.254.169.254/latest/meta-data/placement/availability-zone

Part D: Explore Instance Details and Metadata

14 In the EC2 console, select your lab-web-server instance. Select your instance and review each tab in the lower panel: Details, Security, Networking, Storage, Status checks. For each tab, note the following:

  • Details — Note the instance ID, AMI ID, instance type, and key pair name
  • Security — Review the assigned security group and its inbound/outbound rules
  • Networking — Note the private IPv4 address, public IPv4 address, subnet ID, and Availability Zone
  • Storage — Note the root volume ID, size, and volume type
  • Status checks — Confirm both the instance status check and system status check are passing

15 Connect to the instance using EC2 Instance Connect. Run the following metadata queries to retrieve instance information from within the instance:

curl http://169.254.169.254/latest/meta-data/ami-id
curl http://169.254.169.254/latest/meta-data/local-ipv4
curl http://169.254.169.254/latest/meta-data/public-ipv4
curl http://169.254.169.254/latest/meta-data/security-groups

Compare the values returned with what you noted in the console tabs.

16 Now use IMDSv2 (Instance Metadata Service version 2), which requires a session token. Run this command to retrieve the instance ID using IMDSv2:

TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") && curl -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/instance-id

IMDSv2 is more secure than IMDSv1 because it uses a session-based token, which mitigates certain types of attacks such as Server-Side Request Forgery (SSRF). AWS recommends using IMDSv2 for all new instances.

Part E: Modify Security Group Rules

17 In the EC2 console, click Security Groups in the left navigation (under Network & Security). Select the lab-web-sg security group.

18 Click the Inbound rules tab, then click Edit inbound rules. Delete the HTTP (port 80) rule and click Save rules.

19 Go back to the browser tab where you opened your web page (http://<public-ip>) and refresh the page. The page should time out or fail to load because port 80 is now blocked.

20 Return to the security group. Click Edit inbound rules and add the HTTP rule back:

  • Type: HTTP
  • Source: 0.0.0.0/0

Click Save rules. Refresh your web page browser tab — it should load again.

21 Add one more inbound rule to the security group:

  • Type: HTTPS
  • Port: 443
  • Source: 0.0.0.0/0

Click Save rules. Note that security groups are stateful — when you allow inbound traffic on a port, the return (outbound) traffic is automatically allowed without needing an explicit outbound rule.

Part F: Create and Attach an EBS Volume

22 In the EC2 console, click Volumes in the left navigation (under Elastic Block Store). Click Create volume.

23 Configure the volume:

  • Volume type: General Purpose SSD (gp3)
  • Size: 10 GiB
  • Availability Zone: Select the same AZ as your running instance (check the instance details if needed)

Click Create volume.

24 Select the new volume, click the Name field (or use Tags), and name it lab-extra-volume.

25 With the volume selected, click Actions > Attach volume. Select your lab-web-server instance and set the device name to /dev/sdf. Click Attach volume.

Important: The EBS volume must be in the same Availability Zone as the instance. EBS volumes cannot be attached across AZs.

26 Connect to your instance via EC2 Instance Connect and run the following commands to format, mount, and use the new volume:

# Verify the new volume appears as a block device
lsblk

# Format the volume with the XFS filesystem
sudo mkfs -t xfs /dev/xvdf

# Note: Linux may rename the device from /dev/sdf to /dev/xvdf.
# If lsblk shows a different name, use that name in the commands below.

# Create a mount point and mount the volume
sudo mkdir /data
sudo mount /dev/xvdf /data

# Write a test file to the volume
sudo sh -c 'echo "Hello from EBS" > /data/test.txt'

# Verify the file
cat /data/test.txt

You should see Hello from EBS printed to the terminal. This data is stored on the EBS volume, which persists independently of the instance — if you stop or terminate the instance, the data on this volume remains intact (as long as the volume is not set to delete on termination).

Part G: Create an EBS Snapshot

27 Back in the EC2 console, go to Volumes. Select the lab-extra-volume and click Actions > Create snapshot.

28 Add a description such as lab-snapshot and click Create snapshot.

29 Navigate to Snapshots in the left navigation (under Elastic Block Store). Wait for the snapshot status to change to Completed.

30 Select the snapshot and explore its details:

  • Size — matches the original volume size (10 GiB)
  • Status — Completed
  • Encryption — whether the snapshot is encrypted

Snapshots are point-in-time copies of EBS volumes stored in Amazon S3. They enable backups, disaster recovery, and cross-AZ or cross-Region volume migration by creating new volumes from snapshots in different locations.

Part H: Clean Up

31 Navigate to Instances. Select lab-web-server, click Instance state > Terminate instance. Confirm the termination.

32 Navigate to Snapshots. Select the lab-snapshot, click Actions > Delete snapshot. Confirm the deletion.

33 Navigate to Volumes. Wait for the instance to fully terminate (the root volume will be deleted automatically). Select lab-extra-volume, click Actions > Delete volume. Confirm the deletion.

34 Navigate to Security Groups. Select lab-web-sg and delete it (you may need to wait until the instance is fully terminated).

Lab Complete

You have explored VPC components, launched an EC2 instance with a user data script, verified web server connectivity, queried instance metadata using IMDSv1 and IMDSv2, modified security group rules to observe their effect on traffic, created and mounted an EBS volume, taken a snapshot for backup, and cleaned up all resources.

Discussion Questions

  1. What role does the security group play in controlling access to your instance?
  2. Why did we need an internet gateway and a public IP for the web server to be accessible?
  3. What would happen if you removed the HTTP inbound rule from the security group?
  4. How is the user data script useful for automating instance configuration?
  5. Why must an EBS volume be in the same Availability Zone as the instance it is attached to, and how can snapshots help overcome this limitation?
  6. Security groups are described as "stateful." What does this mean in practice, and how does it differ from Network ACLs, which are "stateless"?
