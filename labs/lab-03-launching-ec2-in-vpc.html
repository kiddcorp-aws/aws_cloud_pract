Lab 3: Launching an EC2 Instance in a VPC

Duration: 15 minutes

Level: Introductory

Prerequisites: Completed Labs 1-2; AWS console access with EC2 and VPC permissions

Lab Objectives

  • Explore the default VPC and its components
  • Review subnets, route tables, and internet gateways
  • Launch an EC2 instance with a security group
  • Connect to the instance using EC2 Instance Connect
  • Terminate the instance and clean up
Important: Use the us-east-1 region for this lab. Ensure you terminate all resources at the end to avoid charges.

Part A: Explore the Default VPC

1 Navigate to VPC in the AWS Management Console. Ensure you are in us-east-1.

2 In the left navigation, click Your VPCs. You should see a default VPC. Note its:

  • VPC ID
  • IPv4 CIDR block (typically 172.31.0.0/16)
  • State (Available)

3 Click Subnets in the left navigation. Note that the default VPC has one subnet per Availability Zone. Each is a public subnet by default.

4 Click Internet gateways. Note that the default VPC has an internet gateway attached, which is what makes the subnets public.

5 Click Route tables. Select the main route table for the default VPC. In the Routes tab, note:

  • A local route for the VPC CIDR (172.31.0.0/16)
  • A route to 0.0.0.0/0 pointing to the internet gateway

Part B: Launch an EC2 Instance

6 Navigate to EC2 and click Launch instance.

7 Configure the instance:

  • Name: lab-web-server
  • AMI: Amazon Linux 2023 (Free tier eligible)
  • Instance type: t2.micro (Free tier eligible)
  • Key pair: Select Proceed without a key pair (we will use EC2 Instance Connect)

8 Under Network settings, click Edit:

  • VPC: Select the default VPC
  • Subnet: Select any available subnet
  • Auto-assign public IP: Enable
  • Security group: Create a new security group named lab-web-sg
  • Add an inbound rule: Type: HTTP, Source: 0.0.0.0/0
  • Keep the default SSH rule (port 22) from anywhere

9 Expand Advanced details and paste the following in the User data field:

#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello from $(hostname -f)</h1><p>AWS Cloud Practitioner Lab 3</p>" > /var/www/html/index.html

10 Leave all other settings as default and click Launch instance.

Part C: Verify and Connect

11 Wait for the instance state to change to Running and the status checks to show 2/2 checks passed (this may take 1-2 minutes).

12 Select the instance and note its Public IPv4 address. Open a new browser tab and navigate to http://<public-ip>. You should see the "Hello from..." web page.

13 Back in the EC2 console, select the instance, click Connect, and use EC2 Instance Connect to open a browser-based terminal. Run:

curl http://169.254.169.254/latest/meta-data/instance-type
curl http://169.254.169.254/latest/meta-data/placement/availability-zone

Part D: Clean Up

14 Select the instance, click Instance state > Terminate instance. Confirm the termination.

15 Navigate to Security Groups, select lab-web-sg, and delete it (you may need to wait until the instance is fully terminated).

Lab Complete

You have explored VPC components, launched an EC2 instance with a user data script, verified web server connectivity, connected via EC2 Instance Connect, and cleaned up all resources.

Discussion Questions

  1. What role does the security group play in controlling access to your instance?
  2. Why did we need an internet gateway and a public IP for the web server to be accessible?
  3. What would happen if you removed the HTTP inbound rule from the security group?
  4. How is the user data script useful for automating instance configuration?
