Lab 2: IAM Users, Groups, and Policies

Duration: 15 minutes

Level: Introductory

Prerequisites: Completed Lab 1; AWS console access with IAM administrative permissions

Lab Objectives

  • Create an IAM user group with specific permissions
  • Create an IAM user and add them to the group
  • Attach AWS managed policies to the group
  • Create a custom inline policy
  • Test the user's permissions by signing in as the new user

Part A: Create an IAM User Group

1 Navigate to IAM in the AWS Management Console.

2 In the left navigation pane, click User groups, then click Create group.

3 Configure the group:

  • Group name: cloud-pract-developers
  • In the Attach permissions policies section, search for and select:
    • AmazonEC2ReadOnlyAccess
    • AmazonS3ReadOnlyAccess
  • Click Create user group

Part B: Create an IAM User

4 In the left navigation pane, click Users, then click Create user.

5 Configure the user:

  • User name: lab-user-01
  • Check Provide user access to the AWS Management Console
  • Select I want to create an IAM user
  • Set a custom password and uncheck "User must create a new password at next sign-in"
  • Click Next

6 On the permissions page:

  • Select Add user to group
  • Check the cloud-pract-developers group
  • Click Next, then Create user

7 On the success page, note the Console sign-in URL and the user credentials. You will need these in Part D.

Part C: Create a Custom Policy

8 Navigate back to User groups and click on cloud-pract-developers.

9 Click the Permissions tab, then Add permissions > Create inline policy.

10 Switch to the JSON tab and paste the following policy, which allows listing IAM users but denies deleting any IAM resources:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "iam:ListUsers",
                "iam:GetUser",
                "iam:ListGroups"
            ],
            "Resource": "*"
        },
        {
            "Effect": "Deny",
            "Action": [
                "iam:Delete*"
            ],
            "Resource": "*"
        }
    ]
}

11 Click Next, name the policy custom-iam-readonly, and click Create policy.

Part D: Test Permissions

12 Open an incognito/private browser window. Navigate to the console sign-in URL noted in step 7.

13 Sign in as lab-user-01. Try the following actions and note the result (Allowed or Denied):

  • Navigate to EC2 > Instances — Can you see instances? (Expected: Yes, read-only)
  • Try to Launch an instance — Does it succeed? (Expected: No, access denied)
  • Navigate to S3 — Can you see buckets? (Expected: Yes, read-only)
  • Try to Create a bucket — Does it succeed? (Expected: No, access denied)
  • Navigate to IAM > Users — Can you see the user list? (Expected: Yes)

Part E: Clean Up

14 Sign out of the lab-user-01 session. Return to your administrator session.

15 Delete the resources created in this lab:

  • Delete user lab-user-01
  • Delete group cloud-pract-developers

Lab Complete

You have successfully created IAM groups and users, attached managed and custom policies, and tested the principle of least privilege by verifying that the user can only perform authorized actions.

Discussion Questions

  1. Why is it better to attach policies to groups rather than directly to individual users?
  2. What happens when a Deny and an Allow statement conflict for the same action?
  3. How does this exercise demonstrate the principle of least privilege?
