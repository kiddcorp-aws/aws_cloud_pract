Lab 2: IAM Users, Groups, and Policies

Duration: 45 minutes

Level: Introductory

Prerequisites: Completed Lab 1; AWS console access with IAM administrative permissions

Lab Objectives

  • Create IAM user groups with different permission levels
  • Create an IAM user and add them to a group
  • Attach AWS managed policies to groups
  • Create custom inline and standalone IAM policies
  • Write a resource-level policy that restricts access to specific S3 buckets
  • Test the user's permissions by signing in as the new user
  • Use the IAM Policy Simulator to validate policy behavior
  • Generate and review IAM Credential Reports and Access Analyzer findings

Part A: Create an IAM User Group

1 Navigate to IAM in the AWS Management Console.

2 In the left navigation pane, click User groups, then click Create group.

3 Configure the group:

  • Group name: cloud-pract-developers
  • In the Attach permissions policies section, search for and select:
    • AmazonEC2ReadOnlyAccess
    • AmazonS3ReadOnlyAccess
  • Click Create user group

Part B: Create an IAM User

4 In the left navigation pane, click Users, then click Create user.

5 Configure the user:

  • User name: lab-user-01
  • Check Provide user access to the AWS Management Console
  • Select I want to create an IAM user
  • Set a custom password and uncheck "User must create a new password at next sign-in"
  • Click Next

6 On the permissions page:

  • Select Add user to group
  • Check the cloud-pract-developers group
  • Click Next, then Create user

7 On the success page, note the Console sign-in URL and the user credentials. You will need these in Part D.

Part C: Create a Custom Policy

8 Navigate back to User groups and click on cloud-pract-developers.

9 Click the Permissions tab, then Add permissions > Create inline policy.

10 Switch to the JSON tab and paste the following policy, which allows listing IAM users but denies deleting any IAM resources:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "iam:ListUsers",
                "iam:GetUser",
                "iam:ListGroups"
            ],
            "Resource": "*"
        },
        {
            "Effect": "Deny",
            "Action": [
                "iam:Delete*"
            ],
            "Resource": "*"
        }
    ]
}

11 Click Next, name the policy custom-iam-readonly, and click Create policy.

Part D: Test Permissions

12 Open an incognito/private browser window. Navigate to the console sign-in URL noted in step 7.

13 Sign in as lab-user-01. Try the following actions and note the result (Allowed or Denied):

  • Navigate to EC2 > Instances — Can you see instances? (Expected: Yes, read-only)
  • Try to Launch an instance — Does it succeed? (Expected: No, access denied)
  • Navigate to S3 — Can you see buckets? (Expected: Yes, read-only)
  • Try to Create a bucket — Does it succeed? (Expected: No, access denied)
  • Navigate to IAM > Users — Can you see the user list? (Expected: Yes)

14 Sign out of the lab-user-01 session and return to your administrator session for the remaining parts.

Part E: Create a Second Group with Different Permissions

15 In the IAM console, click User groups in the left navigation pane, then click Create group.

16 Configure the group:

  • Group name: cloud-pract-admins
  • In the Attach permissions policies section, search for and select AdministratorAccess
  • Click Create user group

Important: The Danger of AdministratorAccess

The AdministratorAccess policy grants unrestricted access to every AWS service and resource in the account. In a production environment, this policy should be assigned to as few users as possible. A compromised account with admin access can lead to data breaches, resource deletion, or significant unexpected charges. Always prefer granting the minimum permissions required for each role.

Part F: Create a Custom IAM Policy from Scratch

17 In the IAM console, click Policies in the left navigation pane, then click Create policy.

18 Click the JSON tab and replace the default content with the following policy. This policy allows all S3 actions, but only on buckets whose names begin with cloud-pract-lab-:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowS3OnSpecificBuckets",
            "Effect": "Allow",
            "Action": "s3:*",
            "Resource": [
                "arn:aws:s3:::cloud-pract-lab-*",
                "arn:aws:s3:::cloud-pract-lab-*/*"
            ]
        },
        {
            "Sid": "AllowBucketListing",
            "Effect": "Allow",
            "Action": "s3:ListAllMyBuckets",
            "Resource": "*"
        }
    ]
}

19 Click Next. On the Review and create page:

  • Policy name: custom-s3-bucket-restrict
  • Description: Allows full S3 access only on buckets starting with cloud-pract-lab-
  • Click Create policy

20 Now attach this policy to the developers group while removing the broader S3 policy. Navigate to User groups and click on cloud-pract-developers.

21 On the Permissions tab, find AmazonS3ReadOnlyAccess. Select it and click Remove. Confirm the removal.

22 Click Add permissions > Attach policies. Search for custom-s3-bucket-restrict, select it, and click Attach policies.

Resource-Level vs. Service-Level Permissions

The original AmazonS3ReadOnlyAccess policy granted read access to all S3 buckets in the account (service-level). The new custom-s3-bucket-restrict policy restricts actions to specific bucket ARNs (resource-level). Resource-level policies are a best practice because they limit the blast radius if credentials are compromised.

Part G: Use the IAM Policy Simulator

23 In the IAM console, scroll down in the left navigation pane and click Policy Simulator (or navigate directly to https://policysim.aws.amazon.com/).

24 In the Users, Groups, and Roles panel on the left, expand Users and select lab-user-01. The simulator will load all policies attached to this user (directly and through groups).

25 Test S3 access to a matching bucket:

  • From the Select service dropdown, choose S3
  • Select the action GetObject
  • Expand Simulation Settings and set the Resource ARN to arn:aws:s3:::cloud-pract-lab-test/file.txt
  • Click Run Simulation
  • Expected result: Allowed

26 Test S3 access to a non-matching bucket:

  • Keep the service as S3 and the action as GetObject
  • Change the Resource ARN to arn:aws:s3:::other-bucket/file.txt
  • Click Run Simulation
  • Expected result: Denied — the resource does not match the policy's ARN pattern

27 Test EC2 terminate access:

  • From the Select service dropdown, choose EC2
  • Select the action TerminateInstances
  • Click Run Simulation
  • Expected result: Denied — the user only has AmazonEC2ReadOnlyAccess

Why the Policy Simulator Matters

The IAM Policy Simulator lets you test the effective permissions of users, groups, and roles without performing real API calls. This is invaluable for troubleshooting "Access Denied" errors, validating new policies before deployment, and auditing existing permissions. It evaluates all attached policies (managed, inline, group-inherited) together, just as AWS does at runtime.

Part H: Explore IAM Credential Report and Access Analyzer

28 In the IAM console, click Credential report in the left navigation pane.

29 Click Download report. Open the downloaded CSV file in a spreadsheet application or text editor. Review the columns and answer these questions:

  • Which users have console passwords enabled?
  • Which users have MFA enabled? (Look at the mfa_active column)
  • Which users have access keys? Are any access keys unused or old?
  • When was each user's password or access key last used?

Security Best Practice

The credential report is a key auditing tool. In production, you should regularly review it to identify users without MFA, unused credentials that should be removed, and access keys that have not been rotated. AWS recommends rotating access keys every 90 days.

30 In the IAM console, click Access Analyzer in the left navigation pane.

31 If an analyzer already exists, click on it. For each finding, note the Resource (which AWS resource is shared), the External principal (who has access), and the Access level (Read, Write, or Full). These findings identify resources shared with external entities.

32 If no analyzer exists, create one:

  • Click Create analyzer
  • Analyzer name: cloud-pract-analyzer
  • Zone of trust: Current account
  • Click Create analyzer
  • Wait a moment for the analyzer to scan your account, then review any findings that appear

33 Check how many findings the Access Analyzer produced and what resource types are involved. Common finding types include: S3 buckets with public access, IAM roles with cross-account trust policies, or KMS keys shared externally. If no findings appear, that means no resources in this account are shared externally.

Part I: Clean Up

34 Delete the IAM user:

  • Navigate to IAM > Users
  • Select lab-user-01 and click Delete. Confirm the deletion.

35 Delete the custom policy:

  • Navigate to IAM > Policies
  • Search for custom-s3-bucket-restrict
  • If the policy is still attached to a group, detach it first by clicking the policy, going to Entities attached, and detaching it from cloud-pract-developers
  • Then select the policy and click Delete. Confirm the deletion.

36 Delete the user groups:

  • Navigate to IAM > User groups
  • Select cloud-pract-developers and click Delete. Confirm the deletion.
  • Select cloud-pract-admins and click Delete. Confirm the deletion.

37 If you created an Access Analyzer in Part H:

  • Navigate to IAM > Access Analyzer
  • Select the cloud-pract-analyzer and click Delete. Confirm the deletion.

Lab Complete

You have successfully created IAM groups with different permission levels, built custom policies with resource-level restrictions, tested permissions both manually and with the IAM Policy Simulator, and explored IAM security auditing tools including Credential Reports and Access Analyzer. These skills are essential for managing secure access in any AWS environment.
