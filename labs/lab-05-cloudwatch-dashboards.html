Lab 5: CloudWatch Dashboards and Alarms

Duration: 45 minutes

Level: Introductory

Prerequisites: Completed Labs 1-4; AWS console access with CloudWatch permissions

Lab Objectives

  • Explore CloudWatch metrics for AWS services
  • Create a custom CloudWatch dashboard
  • Add metric widgets to the dashboard
  • Create a CloudWatch alarm with SNS notification
  • Publish custom metrics using the AWS CLI
  • Create and explore CloudWatch Log groups and log events
  • Configure alarms on custom metrics
  • Explore CloudTrail event history with filters

Part A: Explore CloudWatch Metrics

1 In the AWS Management Console, type CloudWatch in the search bar and click CloudWatch from the results. Ensure you are in us-east-1.

2 In the left navigation, click Metrics > All metrics. Explore the available metric namespaces. You may see:

  • EC2 — Instance metrics (if any instances are running)
  • S3 — Bucket-level metrics
  • Billing — Estimated charges (available in us-east-1 only)

If no namespaces appear, it may take a few minutes for metrics to populate. Try refreshing the page. At minimum, you should see the Billing namespace if you are in us-east-1.

3 If the Billing namespace is available, click into it, then Total Estimated Charge, and select the EstimatedCharges metric for USD. Observe the graph.

Part B: Create a Dashboard

4 In the left navigation, click Dashboards, then Create dashboard.

5 Name the dashboard cloud-pract-lab-dashboard and click Create dashboard.

6 In the Add widget dialog, select Line chart, then Metrics. Choose a metric to display:

  • If you have running EC2 instances: select EC2 > Per-Instance Metrics > CPUUtilization
  • If no instances are running: select Billing > Total Estimated Charge > EstimatedCharges

Click Create widget.

7 Click Add widget again. This time select Number type. Choose EC2 > Per-Instance Metrics > NetworkIn (or Billing > EstimatedCharges if no instances are running). This creates a single-value display.

8 Click Save to save the dashboard.

Part C: Create a CloudWatch Alarm

9 In the left navigation, click Alarms > All alarms, then Create alarm.

10 Click Select metric. Navigate to Billing > Total Estimated Charge and select the EstimatedCharges metric in USD. Click Select metric.

11 Configure the alarm:

  • Statistic: Maximum
  • Period: 6 hours
  • Threshold type: Static
  • Whenever EstimatedCharges is: Greater than 10 (USD)

12 For Notification:

  • Select In alarm
  • Create a new SNS topic named billing-alerts
  • Enter your email address
  • Click Create topic

13 Name the alarm billing-over-10-usd and click Create alarm.

Note: Check your email and confirm the SNS subscription for the alarm to send notifications.

Part D: Explore CloudTrail

14 Type CloudTrail in the console search bar and click CloudTrail from the results.

15 Click Event history in the left navigation. Review recent API events. You should see events from the resources you created in this lab and previous labs. Note:

  • Event name — The API call that was made (e.g., CreateBucket, RunInstances)
  • User name — The IAM identity that made the call
  • Event time — When it happened
  • Source IP address — Where the call came from

Part E: Publish Custom Metrics with CloudShell

16 Open CloudShell by clicking the terminal icon in the top navigation bar of the AWS Management Console. Wait for the environment to initialize (you will see a $ command prompt when ready, typically 15-30 seconds).

17 Publish a custom metric using the AWS CLI. Run the following command:

aws cloudwatch put-metric-data --namespace "LabCustomMetrics" --metric-name "PageViews" --value 100 --unit Count

18 Publish a few more data points with different values to create a visible trend:

aws cloudwatch put-metric-data --namespace "LabCustomMetrics" --metric-name "PageViews" --value 250 --unit Count
aws cloudwatch put-metric-data --namespace "LabCustomMetrics" --metric-name "PageViews" --value 75 --unit Count

19 In the CloudWatch console, go to Metrics > All metrics. Find the LabCustomMetrics namespace in the custom namespaces section. Click into it and select the PageViews metric to view the graph.

20 Navigate to Dashboards and open cloud-pract-lab-dashboard. Click Add widget, select Number type, and add the LabCustomMetrics > PageViews metric. Click Create widget, then Save the dashboard.

Part F: CloudWatch Logs

21 Navigate to CloudWatch > Logs > Log groups in the left navigation. Note how many log groups exist and their names (e.g., CloudTrail or VPC flow logs). If none exist, that is expected for a new account.

22 Click Create log group. Configure the following:

  • Log group name: /lab/test-logs
  • Retention setting: 1 day

Click Create.

23 Switch to CloudShell and create a log stream within the new log group:

aws logs create-log-stream --log-group-name "/lab/test-logs" --log-stream-name "test-stream"

24 Push a log event to the stream:

aws logs put-log-events --log-group-name "/lab/test-logs" --log-stream-name "test-stream" --log-events timestamp=$(date +%s000),message="Lab 5 test log entry - Hello from CloudWatch Logs"

Note: This command uses bash syntax and must be run in CloudShell (not a local Windows terminal).

25 Go back to the CloudWatch console. Click on the /lab/test-logs log group, then click on the test-stream log stream. Verify you can see the log entry you just pushed.

Key Concept: CloudWatch Logs centralize logging from EC2 instances, Lambda functions, containers, API Gateway, and many other AWS services. The CloudWatch Logs agent or SDK can push application logs from your workloads into CloudWatch for centralized analysis and monitoring.

Part G: CloudWatch Alarms — Additional Configuration

26 Navigate to CloudWatch > Alarms > All alarms, then click Create alarm.

27 Click Select metric. Navigate to LabCustomMetrics > Metrics with no dimensions and select the PageViews metric. Click Select metric.

28 Configure the alarm conditions:

  • Statistic: Average
  • Period: 1 minute
  • Threshold type: Static
  • Whenever PageViews is: Greater than 200

29 For Notification, select In alarm and choose the existing SNS topic billing-alerts. Name the alarm high-page-views and click Create alarm.

30 In CloudShell, push a high metric value to trigger the alarm:

aws cloudwatch put-metric-data --namespace "LabCustomMetrics" --metric-name "PageViews" --value 500 --unit Count

31 Wait 1-2 minutes, then refresh the Alarms page. The high-page-views alarm should transition to the In alarm state. Click on the alarm to view the History tab and observe the state change events.

Key Concept: CloudWatch alarms can trigger more than just notifications. They can also trigger Auto Scaling actions (scale out/in), EC2 actions (stop, terminate, reboot), and Systems Manager actions. This makes alarms a core building block for automated operational responses.

Part H: CloudTrail — Deeper Exploration

32 Navigate to CloudTrail > Event history. Clear any existing filters first, then use the Lookup attributes filter dropdown to explore events by source:

  • Filter by Event source: s3.amazonaws.com — observe all S3 API calls made during your labs
  • Filter by Event source: iam.amazonaws.com — observe IAM API calls from Lab 2
  • Filter by User name: your IAM user — see all actions you have performed

33 Click on a specific event to view the full event details in JSON format. Identify the following fields in the event record:

  • userIdentity — Who made the call?
  • eventName — What action was performed?
  • eventTime — When did it happen?
  • sourceIPAddress — From which IP address?
Key Concept: CloudTrail provides the audit trail required for compliance and security investigations. Every API call made to AWS is recorded, including the identity of the caller, the time, the source IP, and the request/response details. This is essential for governance, compliance, and forensic analysis.

Part I: Clean Up

34 Delete the resources created in this lab:

  • Delete the CloudWatch alarm billing-over-10-usd (Alarms > select alarm > Actions > Delete)
  • Delete the CloudWatch alarm high-page-views
  • Delete the CloudWatch dashboard cloud-pract-lab-dashboard (Dashboards > select > Delete)
  • Delete the log group /lab/test-logs (Logs > Log groups > select > Actions > Delete)
  • Optionally delete the SNS topic billing-alerts (Navigate to SNS > Topics)
Note: The custom metric namespace LabCustomMetrics will automatically expire after 15 months of no new data being published to it. There is no manual deletion step required.

Lab Complete

You have explored CloudWatch metrics, created a custom dashboard, configured billing and custom metric alarms with SNS notifications, published custom metrics via the CLI, created and explored CloudWatch Log groups, and performed deeper CloudTrail event analysis for auditing and compliance purposes.
