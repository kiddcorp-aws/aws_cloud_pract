AWS Database Services

Managed databases for every workload

We begin with AWS database services. AWS offers purpose-built databases for different use cases. Understanding when to use which database is a key exam topic.

AWS Database Offerings

AWS provides purpose-built databases optimized for specific use cases rather than a one-size-fits-all approach.

Relational

  • Amazon RDS
  • Amazon Aurora

NoSQL / In-Memory

  • Amazon DynamoDB
  • Amazon ElastiCache

Analytics

  • Amazon Redshift
  • Amazon Athena
AWS offers more than 15 purpose-built database engines. For the Cloud Practitioner exam, focus on RDS, Aurora, DynamoDB, Redshift, and ElastiCache. Each is designed for a specific access pattern and data model.

Relational vs NoSQL Databases

Aspect Relational (SQL) NoSQL
Data Model Tables with rows and columns Key-value, document, graph, wide-column
Schema Fixed schema, predefined Flexible, schema-on-read
Scaling Vertical (scale up) Horizontal (scale out)
Transactions Strong ACID compliance Varies; eventual consistency common
Use Cases ERP, CRM, financial systems Gaming, IoT, real-time apps
AWS Service Amazon RDS, Aurora Amazon DynamoDB
This comparison helps students decide which database type to recommend. Relational databases enforce structure and are ideal for complex queries and transactions. NoSQL databases offer flexibility and horizontal scalability for high-throughput workloads.

Amazon RDS

Relational Database Service

Amazon RDS is a managed relational database service that handles provisioning, patching, backups, and recovery -- freeing you to focus on your application.

Supported Engines

  • MySQL
  • PostgreSQL
  • MariaDB
  • Oracle
  • SQL Server
  • Amazon Aurora

Key Features

  • Automated backups
  • Multi-AZ deployments
  • Read Replicas
  • Encryption at rest and in transit
  • Automated patching
RDS removes the undifferentiated heavy lifting of database administration. You choose the engine and instance size; AWS handles the rest. Six engines are supported, with Aurora being AWS's own cloud-native option.

Amazon Aurora

Cloud-Native Relational Database

Amazon Aurora is MySQL and PostgreSQL compatible, offering up to 5x the throughput of MySQL and 3x the throughput of PostgreSQL -- at a fraction of the cost of commercial databases.

Performance and Reliability

  • Storage auto-scales up to 128 TB
  • 6 copies of data across 3 AZs
  • Continuous backups to S3
  • Sub-10ms read replica lag

Aurora Serverless

  • On-demand, auto-scaling configuration
  • Pay per second of usage
  • Ideal for variable workloads
  • No capacity planning required
Aurora is AWS's flagship relational database. It is fully managed and compatible with MySQL and PostgreSQL, so existing applications can migrate with minimal changes. Aurora Serverless v2 is particularly useful for dev/test environments and variable workloads.

Multi-AZ Deployments and Read Replicas

Multi-AZ Deployments

Purpose: High availability and failover

  • Synchronous replication to standby
  • Automatic failover on failure
  • Same endpoint after failover
  • Standby is NOT used for reads

Read Replicas

Purpose: Performance and scalability

  • Asynchronous replication
  • Up to 15 replicas (Aurora)
  • Can be in different regions
  • Offloads read traffic from primary

Multi-AZ is for availability. Read Replicas are for performance.

This is a commonly tested distinction. Multi-AZ provides a standby instance for automatic failover -- it does not serve read traffic. Read Replicas handle read-heavy workloads by distributing queries. You can use both together.

Amazon DynamoDB

Managed NoSQL Database

Amazon DynamoDB is a fully managed, serverless, key-value and document database delivering single-digit millisecond performance at any scale.

Serverless

No servers to provision or manage. Automatically scales throughput capacity.

Performance

Single-digit millisecond latency. Built-in caching with DAX.

Reliability

Data replicated across 3 AZs. 99.999% SLA with Global Tables.

DynamoDB is one of AWS's most popular services. It is fully serverless -- you simply create a table and start writing data. It handles capacity management, patching, and replication automatically. Global Tables enable multi-region, multi-active replication.

DynamoDB Core Concepts

Data Model

  • Tables -- collection of items
  • Items -- collection of attributes (like a row)
  • Attributes -- fundamental data elements (like a column)

Primary Keys

  • Partition Key -- single attribute; determines data distribution
  • Composite Key -- partition key + sort key; enables range queries

Capacity Modes: On-Demand (pay-per-request, no planning) or Provisioned (predictable workloads, cost-effective with auto-scaling).

Understanding DynamoDB's data model is essential. Unlike relational databases, DynamoDB does not require a fixed schema beyond the primary key. Each item can have different attributes. The partition key determines which physical partition stores the data.

Amazon Redshift

Cloud Data Warehouse

Amazon Redshift is a fast, fully managed, petabyte-scale data warehouse that makes it simple and cost-effective to analyze all your data using standard SQL.

Key Characteristics

  • Columnar storage format
  • Massively parallel processing (MPP)
  • Scales to petabytes
  • Standard SQL queries

Use Cases

  • Business intelligence
  • Historical data analytics
  • Reporting dashboards
  • Data lake integration
Redshift is purpose-built for analytics, not transactional workloads. It uses columnar storage for efficient aggregation queries. Redshift Serverless removes the need to manage clusters. Redshift Spectrum can query data directly in S3 without loading it.

Amazon ElastiCache

In-Memory Caching

Amazon ElastiCache provides a fully managed in-memory data store compatible with Redis or Memcached, delivering sub-millisecond response times.

Redis

  • Rich data structures
  • Persistence and replication
  • Pub/Sub messaging
  • Multi-AZ with auto-failover

Memcached

  • Simple key-value caching
  • Multi-threaded architecture
  • No persistence
  • Horizontal scaling

Use ElastiCache to reduce database load and accelerate application performance.

ElastiCache sits between your application and database, caching frequently accessed data. Redis is the more feature-rich option with persistence and replication. Memcached is simpler and better for pure caching scenarios. Common use cases include session stores, leaderboards, and real-time analytics.

Content Delivery and DNS

Accelerating content and managing domains

Now we move to content delivery and DNS services. CloudFront and Route 53 work together to deliver content fast and route users to the right resources.

Amazon CloudFront

Content Delivery Network

Amazon CloudFront is a global CDN that securely delivers data, videos, applications, and APIs to users with low latency and high transfer speeds.

How It Works

  • Content cached at 450+ edge locations worldwide
  • Users routed to nearest edge location
  • Origin can be S3, EC2, ALB, or custom
  • Supports static and dynamic content

Key Benefits

  • Reduced latency globally
  • DDoS protection via AWS Shield
  • HTTPS and custom SSL certificates
  • Real-time metrics and logging
CloudFront caches content at edge locations close to users. When a user requests content, CloudFront serves it from the nearest edge location. If the content is not cached, CloudFront retrieves it from the origin and caches it for subsequent requests. This dramatically reduces latency.

Amazon Route 53

DNS and Domain Management

Amazon Route 53 is a highly available, scalable DNS web service that provides domain registration, DNS routing, and health checking.

Routing Policy Description Use Case
Simple Route to a single resource Basic websites
Weighted Distribute traffic by percentage A/B testing, blue/green
Latency Route to lowest-latency region Global applications
Failover Active-passive failover Disaster recovery
Geolocation Route based on user location Localized content, compliance
Route 53 is named after TCP/UDP port 53, the DNS port. It offers 100% availability SLA. Each routing policy serves a specific purpose. Weighted routing is great for canary deployments. Failover routing integrates with health checks for automatic DR.

Route 53 Health Checks

Health Check Types

  • Endpoint -- monitor an IP or domain
  • Calculated -- combine multiple health checks
  • CloudWatch Alarm -- based on CloudWatch metric status

How It Works

  • Health checkers in multiple regions
  • Configurable intervals (10s or 30s)
  • Threshold-based pass/fail
  • Integrates with failover routing

Key Concept: Health checks enable automatic DNS failover. If a primary endpoint becomes unhealthy, Route 53 automatically routes traffic to a healthy secondary endpoint.

Health checks are critical for building resilient architectures. They monitor endpoints at regular intervals from distributed locations. If an endpoint fails the health check threshold, Route 53 stops routing traffic to it. This is foundational for active-passive DR patterns.

Monitoring and Management

Visibility, governance, and operational excellence

Monitoring and management services are essential for operating workloads in the cloud. AWS provides a comprehensive set of tools for observability, auditing, and optimization.

AWS Well-Architected Framework

Best practices for designing reliable, secure, efficient, and cost-effective cloud systems.

PillarFocus
Operational ExcellenceRun and monitor systems to deliver business value
SecurityProtect information and systems through risk mitigation
ReliabilityEnsure workloads perform correctly and consistently
Performance EfficiencyUse computing resources efficiently as demand changes
Cost OptimizationAvoid unnecessary costs and optimize spending
SustainabilityMinimize environmental impacts of cloud workloads
The Well-Architected Framework has six pillars as of 2024 -- Sustainability was the most recent addition. The exam tests knowledge of all six pillars and their key principles. AWS offers a Well-Architected Tool in the console to review workloads against these pillars.

Amazon CloudWatch

Monitoring and Observability

Amazon CloudWatch collects and tracks metrics, collects and monitors log files, and sets alarms -- providing system-wide visibility into resource utilization, application performance, and operational health.

Core Capabilities

  • Metrics -- CPU, network, disk, custom
  • Logs -- centralized log collection
  • Alarms -- threshold-based notifications
  • Dashboards -- visual monitoring

Advanced Features

  • CloudWatch Events / EventBridge
  • Anomaly Detection (ML-based)
  • Container Insights
  • Application Insights
CloudWatch is the central monitoring service for AWS. Every AWS service sends metrics to CloudWatch. You can create alarms that trigger SNS notifications or Auto Scaling actions. Dashboards provide a single pane of glass for operational monitoring. CloudWatch Logs Insights allows querying log data with a purpose-built query language.

CloudWatch Alarms in Action

CloudWatch Alarms watch a single metric over a specified time period and perform one or more actions based on the metric value relative to a threshold.

Alarm States

  • OK -- metric within threshold
  • ALARM -- metric breached threshold
  • INSUFFICIENT_DATA -- not enough data

Alarm Actions

  • Send SNS notification
  • Trigger Auto Scaling policy
  • Stop, terminate, or reboot EC2
  • Execute Systems Manager action
Amazon SNS (Simple Notification Service) delivers alarm notifications. You create a topic, subscribe an email or SMS endpoint, confirm the subscription, and CloudWatch sends alerts to all subscribers when an alarm triggers. CloudWatch Alarms are essential for proactive operations. A common pattern is setting a CPU utilization alarm that triggers Auto Scaling to add instances when demand spikes. SNS is the notification backbone — you create a topic, subscribe endpoints (email, SMS, Lambda), and any alarm or event can publish to that topic. In the lab, students will create an SNS topic and subscribe their email to receive alarm notifications. You can also create composite alarms that combine multiple alarm states.

AWS CloudTrail

API Activity Logging

AWS CloudTrail records API calls made on your account, providing a complete audit trail of who did what, when, and from where.

What It Captures

  • Identity of the API caller
  • Time of the API call
  • Source IP address
  • Request and response details

Use Cases

  • Security analysis and auditing
  • Compliance and governance
  • Operational troubleshooting
  • Incident investigation
Exam Tip: CloudTrail answers "who did what?" while CloudWatch answers "how is the system performing?" These are distinct but complementary services. CloudTrail is enabled by default on all AWS accounts, recording the last 90 days of management events at no charge. For long-term retention, you create a trail that delivers logs to an S3 bucket. CloudTrail is essential for compliance frameworks like PCI DSS, HIPAA, and SOC.

AWS Trusted Advisor

Best Practice Recommendations

AWS Trusted Advisor inspects your AWS environment and provides real-time recommendations across five categories to help you follow AWS best practices.

CategoryWhat It Checks
Cost OptimizationIdle resources, underutilized instances
PerformanceOverutilized resources, service limits approaching
SecurityOpen ports, missing MFA on root, public S3 buckets
Fault ToleranceMulti-AZ and backup configurations
Service LimitsUsage against service quotas
Trusted Advisor is like having an AWS expert review your account. Basic checks are available to all accounts. Business and Enterprise Support plans unlock the full set of checks. Common exam questions ask about which category a specific recommendation falls into.

AWS Config

Resource Configuration Tracking

AWS Config continuously monitors and records your AWS resource configurations, enabling you to assess, audit, and evaluate compliance against desired configurations.

Key Capabilities

  • Configuration history for resources
  • Configuration snapshots
  • Change notifications via SNS
  • Relationships between resources

Config Rules

  • Managed rules (pre-built by AWS)
  • Custom rules (Lambda-based)
  • Automatic compliance evaluation
  • Remediation actions
Key Distinction: CloudTrail records API activity (who did what). Config records resource state (what changed and whether it is compliant). AWS Config provides a detailed view of the configuration of AWS resources. It enables compliance-as-code through Config Rules. For example, you can create a rule that checks whether all S3 buckets have encryption enabled and automatically remediate non-compliant resources.

Orchestration and Deployment

Infrastructure as Code and managed platforms

The final section covers how to deploy and manage infrastructure at scale. CloudFormation and Elastic Beanstalk represent two different approaches to deployment automation.

AWS CloudFormation

Infrastructure as Code

AWS CloudFormation lets you model and provision your entire cloud infrastructure using JSON or YAML templates. Resources are deployed as a stack that can be created, updated, and deleted as a single unit.

Core Concepts

  • Templates -- define resources in JSON/YAML
  • Stacks -- deployed instance of a template
  • Change Sets -- preview changes before applying
  • Drift Detection -- detect manual changes

Benefits

  • Repeatable, consistent deployments
  • Version-controlled infrastructure
  • Automatic rollback on failure
  • Dependency management
CloudFormation is the foundational IaC service on AWS. You describe your desired state in a template, and CloudFormation figures out how to create or update resources to match. If a stack creation fails, CloudFormation automatically rolls back all resources to prevent partial deployments.

AWS Elastic Beanstalk

Platform as a Service

AWS Elastic Beanstalk is a PaaS that deploys and manages applications automatically -- handling capacity provisioning, load balancing, auto-scaling, and health monitoring.

Supported Platforms

  • Java, .NET, Node.js, Python
  • Ruby, Go, PHP
  • Docker containers
  • Custom platforms

What Beanstalk Manages

  • EC2 instances and Auto Scaling
  • Load balancer configuration
  • Health monitoring
  • Platform updates and patching
You upload your code; Beanstalk handles everything else. You retain full control of the underlying resources. Elastic Beanstalk is the fastest way to deploy a web application on AWS. It creates and manages the underlying infrastructure for you. Unlike a true PaaS where you cannot access the underlying resources, Beanstalk gives you full access to the EC2 instances, load balancers, and other resources it creates.

CloudFormation vs Elastic Beanstalk

Aspect CloudFormation Elastic Beanstalk
Approach Infrastructure as Code Platform as a Service
Scope Any AWS resource Web applications
Control Full granular control Managed with some customization
Audience DevOps, infrastructure teams Developers focused on code
Learning Curve Steeper -- requires template knowledge Gentler -- upload code and go
Cost No additional charge No additional charge
This is a useful comparison for the exam. CloudFormation gives you complete control but requires more expertise. Beanstalk abstracts away infrastructure concerns for web developers. Both services are free -- you only pay for the underlying resources they create. An interesting detail: Beanstalk uses CloudFormation under the hood.

Key Takeaways

  • RDS and Aurora provide managed relational databases; DynamoDB is the go-to managed NoSQL service; Redshift handles data warehousing; ElastiCache provides in-memory caching.
  • CloudFront accelerates content delivery via edge locations; Route 53 provides DNS routing with multiple policies and health checks.

  • CloudFormation provides IaC for any AWS resource; Elastic Beanstalk provides a PaaS experience for web applications.
