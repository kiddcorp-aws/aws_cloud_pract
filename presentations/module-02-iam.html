IAM Fundamentals

The Foundation of AWS Security

Let us begin with the fundamentals of IAM. Understanding these core concepts is critical before we move into policies, roles, and best practices.

What is AWS IAM?

Identity and Access Management

IAM enables you to securely control who (authentication) can do what (authorization) in your AWS account.

Global Service

IAM is not region-specific. Users, groups, and roles are available across all AWS regions.

Free to Use

There is no charge for IAM. You only pay for the AWS resources your users consume.

Fine-Grained Control

Control access down to specific API actions on specific resources under specific conditions.

IAM is one of the first services you interact with when setting up an AWS account. It is a global service, meaning an IAM user created in one region is available everywhere. Importantly, IAM itself is free. You are only charged for the resources that your users access. IAM provides extremely fine-grained control, letting you define exactly which API calls a principal can make on which resources.

Root Account vs. IAM Users

Root Account

  • Created when you first set up your AWS account
  • Has unrestricted access to all resources
  • Cannot be limited by IAM policies
  • Required for a few specific tasks (closing account, changing support plan)

IAM Users

  • Created within IAM for individual people or applications
  • Permissions are controlled through policies
  • Can have console access, programmatic access, or both
  • Follow least privilege principle

Critical Rule: Never use the root account for day-to-day tasks. Lock it away, enable MFA, and create IAM users instead.

The root account is the email address you used to create your AWS account. It has full, unrestricted access and cannot be limited by IAM policies. This makes it extremely dangerous for everyday use. Best practice is to lock the root account with MFA, create an IAM admin user for daily operations, and only use root for the handful of tasks that require it, such as closing the account or changing the support plan.

IAM Principals

An IAM principal is an entity that can make requests to AWS services.

Users

Represent a person or application that interacts with AWS. Each user has unique credentials.

  • Console password
  • Access keys (CLI/API)

Groups

A collection of users. Policies attached to a group apply to all members.

  • Simplifies permission management
  • Cannot be nested

Roles

An identity with permissions that can be assumed temporarily by users, applications, or services.

  • No permanent credentials
  • Temporary security tokens
There are three main types of IAM principals. Users are for individual people or applications and have long-term credentials. Groups let you manage permissions collectively, so if you have a team of developers, you attach a policy to the Developers group rather than to each user individually. Groups cannot be nested inside other groups. Roles are unique to AWS. They have no long-term credentials. Instead, when a role is assumed, AWS issues temporary security tokens. This is the preferred approach for granting permissions to services like EC2 or Lambda.

IAM Policies and Permissions

Defining Who Can Do What

Now that we understand the principals, let us look at how permissions work. IAM policies are the mechanism through which you grant or deny access to AWS resources.

IAM Policies

JSON-Based Permission Documents

IAM policies are JSON documents that define permissions. They specify which actions are allowed or denied on which AWS resources.

Key Elements of a Policy Statement:

  • Effect -- Allow or Deny
  • Action -- The API calls permitted (e.g., s3:GetObject)
  • Resource -- The specific AWS resource (ARN)
  • Condition -- Optional constraints (IP, time, MFA)

Evaluation Logic:

  • By default, all requests are denied (implicit deny)
  • An explicit Allow overrides the default deny
  • An explicit Deny always overrides any Allow
IAM policies are written in JSON and attached to users, groups, or roles. Each policy contains one or more statements. The Effect is either Allow or Deny. The Action specifies which API calls are covered. The Resource identifies the specific AWS resources using Amazon Resource Names, or ARNs. Conditions are optional and let you add context-based restrictions. The evaluation logic is important: everything is denied by default, an explicit Allow opens access, but an explicit Deny always wins over any Allow.

Example IAM Policy


{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-app-bucket",
                "arn:aws:s3:::my-app-bucket/*"
            ]
        },
        {
            "Effect": "Deny",
            "Action": "s3:DeleteObject",
            "Resource": "arn:aws:s3:::my-app-bucket/*",
            "Condition": {
                "Bool": {
                    "aws:MultiFactorAuthPresent": "false"
                }
            }
        }
    ]
}

This policy: Allows reading objects and listing the bucket, but denies deleting objects unless MFA is active.

Here is a concrete example. The first statement allows the user to list the bucket and get objects from it. The second statement denies the ability to delete objects unless MFA authentication is present. Notice the Condition block, which is a powerful feature for adding context-based controls. The Version field should always be 2012-10-17, which is the current policy language version.

Types of IAM Policies

Type Description Use Case
AWS Managed Pre-built by AWS, updated automatically Common use cases (ReadOnlyAccess, PowerUserAccess)
Customer Managed Created and maintained by you Organization-specific requirements
Inline Embedded directly in a user, group, or role One-off, tightly coupled permissions

Best Practice: Prefer AWS Managed policies when possible, use Customer Managed policies for custom needs, and avoid Inline policies unless strictly necessary.

AWS provides over 1,000 managed policies covering common access patterns. These are maintained by AWS and updated as new services launch. Customer managed policies give you full control and can be versioned. Inline policies are embedded directly in a principal and cannot be reused. They are generally discouraged because they are harder to manage and audit, but they are useful when you need a strict one-to-one relationship between a policy and a principal.

Principle of Least Privilege

Grant Only the Permissions Required

Every user, group, and role should have only the permissions necessary to perform their intended tasks -- nothing more.

How to Implement

  • Start with zero permissions
  • Add permissions incrementally as needed
  • Use IAM Access Analyzer to identify unused permissions
  • Review and remove unnecessary permissions regularly

Why It Matters

  • Limits blast radius of compromised credentials
  • Reduces accidental resource modifications
  • Required for compliance (SOC 2, PCI-DSS, HIPAA)
  • Core AWS Well-Architected principle

"When in doubt, leave it out."

Least privilege is the single most important security principle in IAM. Many organizations make the mistake of granting AdministratorAccess broadly. This creates enormous risk. Instead, start with no permissions and add only what is needed. IAM Access Analyzer can generate policies based on actual usage, helping you right-size permissions. This principle is also heavily tested on the Cloud Practitioner exam.

IAM Roles

Temporary Credentials for Services and Users

Roles are one of the most powerful and frequently misunderstood features of IAM. Let us dive deeper into how they work and when to use them.

What Are IAM Roles?

Temporary, Assumable Identities

A role is an IAM identity with specific permissions that can be assumed by trusted entities. Roles issue temporary security credentials via the AWS Security Token Service (STS).

Roles vs. Users:

  • No long-term credentials (no passwords or access keys)
  • Credentials expire automatically
  • Can be assumed by multiple entities

Two Policy Types on a Role:

  • Trust Policy -- who can assume the role
  • Permission Policy -- what the role can do
Think of a role as a hat you can put on temporarily. When you assume a role, you get temporary credentials with the permissions defined in that role. When the credentials expire, access is revoked. Every role has two parts: a trust policy that defines who is allowed to assume the role, and a permissions policy that defines what the role can do. This separation is a key security design.

Common Role Use Cases

EC2 Instance Roles

Attach a role to an EC2 instance so applications running on it can access AWS services without embedding access keys.

  • Instance profile delivers credentials
  • Credentials rotate automatically

Lambda Execution Roles

Every Lambda function must have an execution role that grants it permission to interact with other AWS services.

  • Defined at function creation
  • Scoped to function needs

Cross-Account Access

Allow users or services in Account A to assume a role in Account B to access its resources.

  • No credential sharing
  • Auditable via CloudTrail
EC2 instance roles are perhaps the most common use case. Instead of putting access keys on an EC2 instance, which is a major security risk, you attach a role. The instance metadata service delivers temporary credentials that rotate automatically. Lambda functions require an execution role to interact with any other AWS service. Cross-account roles are essential in multi-account architectures. They let you grant access across accounts without sharing credentials.

Service-Linked Roles

Predefined Roles for AWS Services

A service-linked role is a unique type of IAM role linked directly to an AWS service. The service defines the permissions, and only that service can assume the role.

Key characteristics:

  • Created automatically by the service when needed
  • Permissions are predefined and cannot be modified
  • Includes all permissions the service requires to call other services on your behalf
  • Examples: Elastic Load Balancing, Amazon RDS, AWS Config

Service-linked roles simplify setup by providing pre-configured permissions that the service needs.

Service-linked roles are a special category. They are created and managed by AWS services. You cannot modify their permissions because the service itself defines exactly what it needs. For example, when you create an Elastic Load Balancer, AWS automatically creates a service-linked role that allows ELB to register and deregister instances, perform health checks, and other necessary actions. You will see these in your IAM console, and it is important to understand that they are normal and expected.

Security Best Practices

Protecting Your AWS Environment

Security is not optional in the cloud. AWS provides the tools, but it is your responsibility to configure them properly. Let us review the critical best practices for IAM.

Multi-Factor Authentication (MFA)

Non-Negotiable: Enable MFA on the root account and all IAM users with console access.

MFA Options

  • Virtual MFA devices -- Authenticator apps (Google Authenticator, Authy)
  • Hardware MFA tokens -- Physical key fobs
  • FIDO2 security keys -- YubiKey and similar devices
  • Passkeys -- Platform authenticators (supported since 2024)

Why MFA Matters

  • Passwords alone can be phished or leaked
  • MFA adds a second verification factor
  • Blocks over 99% of account compromise attacks
  • Can be required via IAM policy conditions
MFA is the single most impactful security measure you can take. AWS supports multiple MFA device types, including virtual authenticator apps, hardware tokens, and FIDO2 security keys. Since 2024, AWS also supports passkeys for root account MFA. You can also enforce MFA in IAM policies using the aws:MultiFactorAuthPresent condition key, ensuring that sensitive operations require MFA even if the user is already logged in.

Credential Management

Password Policies

  • Set minimum length and complexity requirements
  • Enable password expiration and prevent reuse
  • Allow users to change their own passwords

Access Key Best Practices

  • Never embed keys in code or commit to version control
  • Use IAM roles instead of access keys when possible
  • Rotate keys regularly and remove unused keys

Warning: Exposed access keys are the number one cause of AWS security breaches. Use tools like AWS Secrets Manager or environment variables instead of hardcoding credentials.

Password policies are configured at the account level and apply to all IAM users. For access keys, the golden rule is to never embed them in code. Leaked access keys are the leading cause of AWS account compromises. Bots continuously scan public repositories like GitHub for exposed keys and can begin exploiting them within minutes. Always prefer roles over access keys, and when you must use keys, rotate them regularly and use AWS Secrets Manager to store them.

Advanced Security Tools

IAM Access Analyzer

  • Identifies resources shared with external entities
  • Generates least-privilege policies based on access activity
  • Validates policies against best practices
  • Provides unused access findings to help right-size permissions

AWS Organizations and SCPs

  • Manage multiple AWS accounts centrally
  • Service Control Policies (SCPs) set permission guardrails across accounts
  • SCPs restrict the maximum permissions available
  • Apply to all users and roles in member accounts (including root)

Key Insight: SCPs do not grant permissions. They define the maximum boundary. You still need IAM policies to allow specific actions within those boundaries.

IAM Access Analyzer is a powerful tool that helps you identify overly permissive policies. It can analyze your CloudTrail logs and generate policies based on what was actually used, which is invaluable for implementing least privilege. AWS Organizations lets you manage multiple accounts and apply Service Control Policies. SCPs act as permission boundaries across entire accounts. An important distinction for the exam: SCPs restrict what is possible, but they do not grant permissions. An IAM policy must still explicitly allow the action.

Identity Federation and SSO

Enterprise Identity Integration

Most organizations already have identity systems in place. AWS provides several mechanisms to integrate those existing identity providers with AWS access, eliminating the need to create separate IAM users for every person.

Identity Federation

Use Existing Identities with AWS

Identity federation allows users to authenticate with an external identity provider (IdP) and receive temporary AWS credentials without needing an IAM user account.

SAML 2.0 Federation

  • Integrate with enterprise directories (Active Directory, Okta, Ping Identity)
  • Single sign-on to AWS Management Console
  • Map directory groups to IAM roles

OpenID Connect (OIDC)

  • Used for web and mobile applications
  • Compatible with providers like Google, Facebook, Amazon Cognito
  • Enables identity federation for application users
Federation is essential in enterprise environments. With SAML 2.0, employees can use their corporate credentials to access AWS through single sign-on. The organization's identity provider authenticates the user and provides a SAML assertion that AWS trusts. OIDC federation is more commonly used for application-level identity, where end users of a web or mobile app need access to AWS resources like S3 or DynamoDB through Amazon Cognito.

AWS IAM Identity Center

Formerly AWS Single Sign-On (SSO) -- Renamed in 2022 to better reflect its expanded capabilities.

Key Features

  • Centralized access management for multiple AWS accounts
  • Built-in user directory or connect to existing IdP
  • One-click access to assigned accounts and applications
  • Permission sets define access levels

Supported Identity Sources

  • IAM Identity Center directory (built-in)
  • Active Directory (AWS Managed or self-managed)
  • External SAML 2.0 identity providers (Okta, Azure AD, Ping)

IAM Identity Center is the recommended approach for managing workforce access to AWS accounts and applications.

IAM Identity Center, formerly AWS SSO, is the recommended way to manage human access to AWS in an organization. It integrates with AWS Organizations and lets you define permission sets that map to IAM roles in each account. Users get a single portal where they can see all accounts and applications they have access to. For the exam, remember that this is the successor to AWS SSO and the preferred solution for workforce identity management.

Amazon Cognito

Authentication and Authorization for Applications

Amazon Cognito provides authentication and authorization for web and mobile applications, enabling secure user sign-up, sign-in, and access control.

User Pools

  • Managed user directory for sign-up and sign-in
  • Supports MFA and social identity providers
  • SAML/OIDC federation with enterprise IdPs
  • Returns JSON Web Tokens (JWTs)

Identity Pools

  • Provide temporary AWS credentials to access AWS services directly
  • Supports both authenticated and guest users
  • Maps users to IAM roles for fine-grained access

User Pools = WHO you are (authentication) | Identity Pools = WHAT you can do (authorization/AWS access)

Amazon Cognito is the go-to service for adding authentication and authorization to web and mobile applications. It is distinct from IAM Identity Center, which is designed for workforce access to the AWS console and AWS accounts. Cognito is purpose-built for application end-users. User Pools handle the authentication side, managing user directories and returning JWTs after sign-in. Identity Pools handle the authorization side, vending temporary AWS credentials so that application users can interact with AWS services directly. The two components can be used independently or together.

AWS Key Management Service (KMS)

Managed Encryption Key Service

AWS KMS lets you create and manage cryptographic keys used to encrypt your data across AWS services.

AWS Managed Keys

Created and managed by AWS services on your behalf. Free to use, automatic rotation.

Customer Managed Keys

You create and control the key policies. Can configure automatic annual rotation.

Integration

Natively integrated with S3, EBS, RDS, Lambda, and 100+ AWS services.

IAM + KMS: Access to encryption keys is controlled through IAM policies and KMS key policies working together.

KMS ties closely to IAM because access to encryption keys is governed by IAM policies and KMS key policies. AWS managed keys are created automatically when you enable encryption on a service. Customer managed keys give you full control, including the ability to define who can use and administer the key. For the Cloud Practitioner exam, you should understand that KMS provides managed encryption and integrates across AWS services.

Key Takeaways

  • IAM is global and free -- it controls authentication and authorization for all AWS services
  • Never use the root account for daily tasks -- lock it with MFA and create IAM users instead
  • Use Groups to assign permissions to users, and Roles for services and cross-account access
  • Policies are JSON documents with Effect, Action, and Resource -- explicit Deny always wins
  • Apply least privilege -- start with no permissions, add only what is needed
  • Enable MFA everywhere -- use IAM Identity Center for enterprise workforce access
Let us review the key points. IAM is the security backbone of AWS. The root account should be locked away. Groups simplify user permission management. Roles are for services and cross-account access. Policies use JSON with Allow and Deny effects, and Deny always takes precedence. Least privilege is the guiding principle. MFA should be enabled on every account. And IAM Identity Center is the modern approach for enterprise workforce access.
