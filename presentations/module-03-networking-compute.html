Amazon VPC

Virtual Private Cloud

We begin with Amazon VPC, which is the networking foundation for nearly everything you deploy on AWS. Without understanding VPC, it is difficult to reason about security, connectivity, and architecture in the cloud.

What Is a VPC?

A Virtual Private Cloud (VPC) is a logically isolated section of the AWS cloud where you can launch AWS resources in a virtual network that you define.

  • You control IP address ranges, subnets, route tables, and gateways
  • Each AWS account includes a default VPC in every Region
  • VPCs are Region-scoped and can span multiple Availability Zones
  • CIDR blocks define the IP address range (e.g., 10.0.0.0/16)
Think of a VPC as your own private data center within AWS. You get full control over the networking environment. The default VPC is provided so new users can launch instances without having to configure networking first, but production workloads typically use custom VPCs with carefully planned CIDR ranges.

VPCs, AZs, and Regions

Region

  • A VPC belongs to a single Region
  • Cannot span Regions
  • Use VPC Peering or Transit Gateway for cross-Region connectivity

Availability Zones

  • A VPC can span all AZs in its Region
  • Subnets are AZ-scoped
  • Deploy across multiple AZs for high availability
VPC = Region-scoped | Subnet = AZ-scoped
This relationship is fundamental. A VPC lives within one Region, but you create subnets inside specific Availability Zones. To build highly available architectures, you deploy resources across subnets in multiple AZs within the same VPC.

Core VPC Components

Subnets

Subdivisions of a VPC's IP range, each in a single AZ. Can be public or private.

Route Tables

Rules that determine where network traffic is directed. Each subnet has one associated route table.

Internet Gateway (IGW)

Allows communication between resources in a VPC and the internet. Horizontally scaled and highly available.

NAT Gateway

Enables private subnet resources to access the internet for outbound traffic while remaining unreachable from inbound.

These four components form the backbone of VPC networking. Subnets segment your network, route tables control traffic flow, the internet gateway provides public internet access, and the NAT gateway lets private resources make outbound connections without exposing them to inbound internet traffic.

Public vs Private Subnets

Public Subnet

  • Route table has a route to an Internet Gateway
  • Resources can have public IP addresses
  • Directly reachable from the internet
  • Use for: load balancers, bastion hosts, public-facing web servers

Private Subnet

  • No route to an Internet Gateway
  • Resources only have private IP addresses
  • Outbound via NAT Gateway if needed
  • Use for: databases, application servers, internal services
Best Practice: Place resources in private subnets by default. Only use public subnets when direct internet access is required.
The distinction between public and private subnets comes down to routing. If a subnet's route table includes a route to an internet gateway, it is public. Most production workloads place application tiers and data tiers in private subnets and only expose a load balancer or bastion in the public subnet.

Security Groups

A Security Group acts as a virtual firewall at the instance level, controlling inbound and outbound traffic.

  • Stateful -- return traffic is automatically allowed
  • Rules are allow-only -- you cannot create deny rules
  • All rules are evaluated before deciding to allow traffic
  • Can reference other security groups as sources
  • Default: allows all outbound, denies all inbound
Security groups are probably the most commonly used network security feature in AWS. Because they are stateful, if you allow inbound traffic on port 443, the response traffic is automatically permitted without a separate outbound rule. This simplifies configuration compared to traditional firewalls.

Network ACLs

A Network Access Control List (NACL) is a firewall at the subnet level that controls traffic entering and leaving a subnet.

  • Stateless -- return traffic must be explicitly allowed
  • Supports both allow and deny rules
  • Rules are evaluated in number order (lowest first)
  • Default NACL allows all inbound and outbound traffic
  • One NACL per subnet; one subnet can have only one NACL
NACLs provide a second layer of defense at the subnet boundary. Because they are stateless, you need to configure both inbound and outbound rules. They are useful for broad deny rules, such as blocking a known malicious IP range at the subnet level.

Security Groups vs NACLs

Feature Security Group Network ACL
Scope Instance (ENI) level Subnet level
State Stateful Stateless
Rule Type Allow only Allow and Deny
Rule Evaluation All rules evaluated Rules in number order
Default Inbound Deny all Allow all
Return Traffic Automatic Must be explicitly allowed
Remember: Security Groups are stateful (SG = Stateful). NACLs are stateless and operate at the subnet boundary.
This comparison table is a frequent topic on the Cloud Practitioner exam. The key differentiators to remember are stateful versus stateless, allow-only versus allow-and-deny, and instance level versus subnet level. Most workloads rely heavily on security groups with NACLs as an additional defense layer.

VPC Connectivity

VPC Peering

  • Private connection between two VPCs
  • Works across accounts and Regions
  • Non-transitive -- each pair needs its own peering
  • Best for a small number of VPC connections

Transit Gateway

  • Hub-and-spoke model for connecting multiple VPCs
  • Supports thousands of VPCs
  • Can connect VPCs and on-premises networks
  • Best for large-scale, complex networking
VPC Peering is straightforward for connecting two VPCs but does not scale well because every pair needs its own peering connection and peering is non-transitive. Transit Gateway solves this with a central hub that all VPCs connect to, greatly simplifying network topology in large organizations.

Hybrid Connectivity

AWS Site-to-Site VPN

  • Encrypted IPsec tunnel over the public internet
  • Connects on-premises network to a VPC
  • Quick to set up -- operational in minutes
  • Lower cost, but variable latency

AWS Direct Connect

  • Dedicated private connection from on-premises to AWS
  • Consistent low latency, higher throughput
  • Takes weeks to establish
  • Higher cost than VPN
VPN = encrypted over internet (quick, variable latency) | Direct Connect = dedicated private line (consistent, higher cost)
Site-to-Site VPN and Direct Connect are the two primary ways to connect an on-premises data center to AWS. VPN is the faster and cheaper option -- you can have an encrypted tunnel running in minutes. However, because the traffic traverses the public internet, latency can vary. Direct Connect provides a dedicated physical connection through an AWS partner facility, delivering consistent performance and higher bandwidth. It is the right choice for workloads that require predictable latency or move large volumes of data, but it takes weeks to provision and costs more. Many organizations use both: Direct Connect as the primary path and VPN as a backup.

VPC Endpoints

VPC Endpoints allow private connectivity from your VPC to supported AWS services without traversing the internet, improving security and reducing data transfer costs.

Gateway Endpoints

  • Support S3 and DynamoDB only
  • Route table entries -- no additional charge
  • Traffic stays within the AWS network

Interface Endpoints (AWS PrivateLink)

  • ENI with a private IP -- supports 100+ AWS services
  • Hourly charge plus per-GB data processing fee
  • Also used to expose your own services to other VPCs
Key takeaway: VPC Endpoints keep traffic within the AWS network, improving security and reducing data transfer costs.
VPC Endpoints remove the need for an internet gateway, NAT device, or VPN connection to reach AWS services. Gateway Endpoints are free and work by adding a route to your route table -- they are available only for S3 and DynamoDB. Interface Endpoints use AWS PrivateLink and create an ENI in your subnet with a private IP address, supporting a much broader set of services but at a per-hour and per-GB cost. For the exam, remember the distinction: Gateway Endpoints are free and limited to S3 and DynamoDB; Interface Endpoints cost money but cover many more services.

Amazon EC2

Elastic Compute Cloud

Now we move to Amazon EC2, the core compute service in AWS. EC2 gives you resizable virtual servers that you can provision in minutes rather than weeks, which was revolutionary when it launched and remains foundational to cloud computing.

What Is Amazon EC2?

Amazon Elastic Compute Cloud (EC2) provides virtual servers -- called instances -- in the cloud, giving you resizable compute capacity on demand.

  • Launch instances in minutes, scale up or down as needed
  • Pay only for the compute time you use
  • Choose from hundreds of instance types optimized for different workloads
  • Full root/admin access to your instances
  • Runs within your VPC for network isolation
EC2 was one of the first AWS services and remains one of the most widely used. It replaces the need to invest in physical hardware upfront. You can experiment, develop, and scale without long-term commitments, paying only for what you consume.

EC2 Instance Families

General Purpose (M, T, A)

Balanced compute, memory, networking — web servers, dev environments

Compute Optimized (C)

High-performance processors — batch processing, HPC, media encoding

Memory Optimized (R, X, z)

Large in-memory data sets — in-memory databases, real-time analytics

Storage Optimized (I, D, H)

High sequential read/write — data warehousing, distributed file systems

Accelerated Computing (P, G, Inf, Trn)

Hardware accelerators (GPUs/FPGAs) — ML training, graphics rendering

AWS organizes instance types into families optimized for different workloads. For the Cloud Practitioner exam, you need to know the general categories and their use cases. You do not need to memorize specific instance specifications, but understanding which family fits which workload is important.

Instance Type Naming

m5.xlarge

m = Instance family (General Purpose)

5 = Generation number

xlarge = Instance size

Sizes: nano, micro, small, medium, large, xlarge, 2xlarge...

Tip: Higher generation numbers generally offer better price-performance. Prefer current-generation instances (e.g., m7i over m5) for new workloads.
The naming convention is straightforward once you understand the pattern. The letter identifies the family, the number is the generation, and the size determines the amount of vCPU, memory, and network capacity. Each step up in size typically doubles the resources and cost.

Amazon Machine Images (AMIs)

An AMI is a template that contains the software configuration (OS, application server, applications) required to launch an instance.

AMI Sources

  • AWS-provided (Amazon Linux, Ubuntu, Windows)
  • AWS Marketplace (third-party)
  • Community AMIs
  • Custom AMIs you create

Key Characteristics

  • Region-specific (can be copied across Regions)
  • Includes root volume and optional data volumes
  • Launch permissions control who can use it
  • Foundation for repeatable, consistent deployments
AMIs are the blueprints for your EC2 instances. Creating custom AMIs from a configured instance is a common practice called golden image baking. This ensures every instance you launch is identical and already has the software and patches you need, reducing configuration drift.

EC2 Pricing Models

Model Commitment Savings Best For
On-Demand None Baseline (no discount) Short-term, unpredictable workloads
Reserved Instances 1 or 3 years Up to 72% Steady-state, predictable usage
Savings Plans 1 or 3 years ($/hr) Up to 72% Flexible across instance families
Spot Instances None Up to 90% Fault-tolerant, flexible workloads
Dedicated Hosts On-Demand or Reserved Varies Licensing, compliance requirements
Spot Instances can be interrupted by AWS with a 2-minute warning. Only use them for workloads that can handle interruptions (batch jobs, CI/CD, data analysis).
Pricing models are a popular exam topic. On-Demand is the most flexible but most expensive. Reserved Instances and Savings Plans trade commitment for discounts. Spot gives the deepest discounts but can be reclaimed. Dedicated Hosts are for specific compliance or licensing needs such as per-socket or per-core software licenses.

EC2 Instance Lifecycle

pending → running → stopping → stopped → terminated

Running State

  • Instance is active and billing applies
  • Can be stopped (EBS-backed) or terminated
  • Can be rebooted without data loss

Stopped vs Terminated

  • Stopped: No compute charges; EBS volumes persist
  • Terminated: Instance is deleted; EBS root volume deleted by default
  • Instance store data is lost on stop or terminate
Understanding the lifecycle matters for both cost management and operations. When you stop an instance, you stop paying for compute but still pay for EBS storage. Terminating is permanent and the instance cannot be restarted. Termination protection can be enabled to prevent accidental deletion.

Amazon EBS Overview

Amazon Elastic Block Store (EBS) provides persistent block-level storage volumes for EC2 instances. Think of it as a virtual hard drive.

Volume Type Category Use Case
gp3 General Purpose SSD Boot volumes, dev/test, virtual desktops
io2 Provisioned IOPS SSD Databases needing sustained IOPS
st1 Throughput Optimized HDD Big data, data warehouses, log processing
sc1 Cold HDD Infrequent access, lowest cost
Key fact: EBS volumes are AZ-scoped and can only attach to instances in the same AZ. Use EBS Snapshots to copy data across AZs or Regions.
EBS is the primary storage mechanism for EC2. gp3 is the default and recommended volume type for most workloads, offering a good balance of price and performance. io2 is for I/O-intensive database workloads. HDD options are cheaper for sequential access patterns but cannot be used as boot volumes.

Elastic IP Addresses

What Are Elastic IPs?

  • Static, public IPv4 addresses you allocate to your account
  • Can be associated and disassociated with instances
  • Persist until you explicitly release them
  • Useful for maintaining a fixed IP after instance replacement

Important Considerations

  • Free when associated with a running instance
  • Charged when allocated but not in use
  • Limited to 5 per Region by default
  • Prefer DNS names and load balancers over Elastic IPs when possible
Cost note: As of February 2024, AWS charges for all public IPv4 addresses, including Elastic IPs -- even when attached to running instances.
Elastic IPs provide a stable endpoint, but AWS now charges for all public IPv4 addresses to encourage IPv6 adoption and efficient IP usage. For most architectures, placing instances behind a load balancer with a DNS name is preferred over assigning Elastic IPs directly.

Serverless Compute: AWS Lambda

AWS Lambda lets you run code without provisioning or managing servers. You supply the code, and AWS handles all the infrastructure, scaling, and availability.

How Lambda Works

  • Event-driven -- triggered by S3, API Gateway, DynamoDB, and more
  • Scales automatically with concurrent executions
  • Pay per request plus compute duration -- no charge when idle

Key Characteristics

  • Supports Python, Node.js, Java, Go, .NET, Ruby, and custom runtimes
  • 15-minute max execution; up to 10 GB memory per function
  • Integrates natively with many AWS services
Lambda is the foundation of serverless architecture on AWS -- no servers to manage, no idle cost.
Lambda vs EC2 is a common exam comparison. EC2 gives you full control over a virtual server that runs continuously (or on a schedule you manage), while Lambda runs code only in response to events and scales automatically. Lambda is ideal for short-duration, event-driven tasks such as processing an uploaded file, handling an API request, or reacting to a database change. EC2 is better suited for long-running processes, applications that need persistent state on disk, or workloads that require a specific operating system configuration. Remember the 15-minute execution limit -- if your workload runs longer than that, Lambda is not the right fit.

High Availability & Scaling

Load Balancing and Auto Scaling

Now that we understand the individual compute and networking components, we need to discuss how to make them resilient and scalable. This section covers the services that help you distribute traffic and automatically adjust capacity.

Elastic Load Balancing

Elastic Load Balancing (ELB) automatically distributes incoming application traffic across multiple targets (EC2 instances, containers, IPs) in one or more AZs.

Application LB (ALB)

  • Layer 7 (HTTP/HTTPS)
  • Path- and host-based routing
  • Best for web applications

Network LB (NLB)

  • Layer 4 (TCP/UDP)
  • Ultra-low latency
  • Millions of requests/sec

Gateway LB (GLB)

  • Layer 3 (IP packets)
  • Third-party appliances
  • Firewalls, IDS/IPS
For the Cloud Practitioner exam, know that ALB is for HTTP/HTTPS traffic with advanced routing, NLB is for high-performance TCP/UDP traffic, and GLB is for deploying third-party network appliances. ALB is the most commonly used type for typical web applications.

EC2 Auto Scaling

How It Works

  • Scale out: Add instances when demand increases
  • Scale in: Remove instances when demand decreases
  • Maintains a desired count of healthy instances
  • Works across multiple AZs for high availability

Scaling Policies

  • Target tracking: Maintain a metric at a target (e.g., CPU at 50%)
  • Step scaling: Scale by amounts based on alarm thresholds
  • Scheduled: Scale at specific times
  • Predictive: ML-based forecasting
Auto Scaling ensures you have the right number of instances at all times -- improving availability and reducing cost.
Auto Scaling is a key differentiator of cloud computing. You define minimum, maximum, and desired capacity. AWS handles the rest. Combined with a load balancer, Auto Scaling enables architectures that can handle traffic spikes without manual intervention and save money during low-traffic periods.

Launch Templates

A Launch Template defines the configuration for an EC2 instance, enabling consistent and repeatable instance launches.

A launch template specifies:
  • AMI ID and instance type
  • Key pair and security groups
  • Network settings (subnet, public IP)
  • Storage (EBS volumes)
  • User data scripts for bootstrapping
  • IAM instance profile
Best Practice: Use launch templates instead of launch configurations. Launch templates support versioning and are required by current-generation Auto Scaling features.
Launch templates are the modern way to define instance configurations. They replaced launch configurations and offer versioning, which means you can iterate on your instance setup without creating entirely new templates. Auto Scaling groups reference launch templates to know how to launch new instances.

Key Takeaways

  • VPC is your isolated network in AWS -- it is Region-scoped with AZ-scoped subnets, and you control all routing and access.
  • Security Groups are stateful instance-level firewalls; NACLs are stateless subnet-level firewalls.
  • EC2 provides resizable virtual servers with multiple instance families optimized for different workload types.
  • EC2 pricing ranges from flexible On-Demand to deeply discounted Spot and Reserved options -- choose based on workload characteristics.
  • EBS provides persistent block storage with volume types tailored to performance and cost needs.
  • ELB and Auto Scaling work together to deliver highly available, automatically scaling architectures.
These are the essential concepts from this module. VPC and EC2 form the backbone of most AWS architectures. Understanding networking isolation, security layers, instance selection, pricing models, and scaling patterns will serve you well on both the exam and in real-world AWS work.
