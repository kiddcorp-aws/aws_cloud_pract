AWS Storage Overview

Block, File, and Object Storage

Let us start with a foundational overview of the three main categories of cloud storage and then map them to the AWS services that implement each type.

Types of Cloud Storage

Block Storage

Data is stored in fixed-size blocks, each with a unique address. The OS treats it like a raw disk.

  • Low latency
  • Ideal for databases
  • Attached to one instance

File Storage

Data is organized in a hierarchical directory structure. Accessed via standard file protocols (NFS, SMB).

  • Shared access
  • Familiar file/folder model
  • Good for home directories

Object Storage

Data is stored as objects with metadata and a unique identifier. Accessed via HTTP/REST APIs.

  • Virtually unlimited scale
  • Rich metadata
  • Ideal for unstructured data
Block storage is like a hard drive: you format it with a file system and mount it. File storage is a managed file system that multiple clients can connect to. Object storage is flat — no hierarchy — and each object includes the data, metadata, and a unique key. Understanding these three types is essential for choosing the right AWS service.

AWS Storage Services at a Glance

Service Type Key Characteristic
Amazon S3 Object Scalable object storage for any data type
Amazon EBS Block Persistent block storage for EC2 instances
Amazon EFS File Elastic NFS file system for Linux workloads
Amazon FSx File Managed file systems (Windows, Lustre, NetApp, OpenZFS)
Storage Gateway Hybrid On-premises to cloud storage bridge
AWS offers a broad set of storage services. Each is designed for specific workload patterns. S3 dominates for object storage and is one of the most important services for the Cloud Practitioner exam. EBS is block storage tied to EC2. EFS and FSx are managed file systems. Storage Gateway connects on-premises environments to AWS storage.

Amazon S3

Simple Storage Service

Amazon S3 is arguably the most important storage service in AWS. It was one of the first AWS services launched and remains central to nearly every AWS architecture. Let us explore it in detail.

What is Amazon S3?

Amazon Simple Storage Service (S3) is an object storage service offering industry-leading scalability, availability, durability, and security.

Core Concepts

  • Buckets — Containers for objects (globally unique name)
  • Objects — Files stored in buckets (up to 5 TB)
  • Keys — Unique identifier for each object within a bucket

Key Properties

  • 99.999999999% (11 nines) durability
  • Virtually unlimited storage
  • Regional service with global namespace
S3 stores data as objects within buckets. A bucket name must be globally unique across all of AWS. Objects consist of the data itself, metadata, and a key. The key is the full path to the object within the bucket. S3 delivers eleven nines of durability, meaning the probability of losing an object is astronomically low.

S3 Bucket Properties and Configuration

Bucket Settings

  • Region — Data stays in the selected region
  • Versioning — Track object versions
  • Logging — Access logs for auditing
  • Tags — Key-value pairs for cost allocation

Access and Security

  • Block Public Access — Enabled by default
  • Bucket Policy — JSON-based resource policy
  • Encryption — Default encryption at rest
  • Object Lock — WORM compliance

S3 buckets block public access by default — you must explicitly enable public access if needed.

When creating a bucket, you choose a region for data residency and latency optimization. Versioning, logging, and encryption can all be configured at bucket creation time or later. Since 2023, S3 encrypts all new objects by default with SSE-S3. The Block Public Access setting is on by default, adding a safety net against accidental data exposure.

S3 Storage Classes

Storage Class Use Case Retrieval Availability
S3 Standard Frequently accessed data Milliseconds 99.99%
Intelligent-Tiering Unknown or changing access Milliseconds 99.9%
Standard-IA Infrequent access, rapid retrieval Milliseconds 99.9%
One Zone-IA Recreatable infrequent data Milliseconds 99.5%
Glacier Instant Retrieval Archive, instant access needed Milliseconds 99.9%
Glacier Flexible Retrieval Archive, minutes to hours OK 1 min to 12 hrs 99.99%
Glacier Deep Archive Long-term archive, rarely accessed 12 to 48 hrs 99.99%
S3 offers seven primary storage classes, each optimized for different access patterns and cost requirements. Standard is the default for frequently accessed data. Intelligent-Tiering automatically moves objects between tiers based on access patterns — ideal when you cannot predict usage. The IA classes have lower storage costs but charge retrieval fees. The Glacier classes are for archival with varying retrieval speeds. Deep Archive is the lowest cost storage option in AWS.

S3 Object Versioning

Versioning preserves, retrieves, and restores every version of every object stored in a bucket.

How It Works

  • Enabled at the bucket level
  • Each object gets a unique Version ID
  • Overwriting creates a new version, not a replacement
  • Deleting adds a delete marker

Important Notes

  • Once enabled, versioning cannot be disabled — only suspended
  • All versions consume storage and incur costs
  • Use lifecycle rules to manage old versions
Versioning is a powerful data protection mechanism. When enabled, every PUT operation creates a new version. DELETE operations do not actually remove the object — they insert a delete marker. You can restore previous versions at any time. Be aware that every version is stored and billed, so combine versioning with lifecycle policies to control costs.

S3 Lifecycle Policies

Lifecycle policies automate the transition of objects between storage classes or schedule their expiration.

Transition Actions

  • Move to Standard-IA after 30 days
  • Move to Glacier after 90 days
  • Move to Deep Archive after 180 days

Expiration Actions

  • Delete objects after a set number of days
  • Delete previous versions after N days
  • Remove incomplete multipart uploads

Lifecycle policies are the primary mechanism for optimizing S3 storage costs over time.

Lifecycle policies allow you to define rules that automatically transition objects to cheaper storage classes as they age, or delete them entirely. You can apply rules to all objects in a bucket or filter by prefix or tags. A common pattern is: Standard for the first 30 days, then IA, then Glacier, then Deep Archive. Combine with versioning to also expire old versions.

S3 Security

Access Control

  • Bucket Policies — JSON resource-based policies for fine-grained access
  • IAM Policies — Identity-based policies for users and roles
  • ACLs — Legacy mechanism, generally not recommended
  • Block Public Access — Account and bucket-level safety

Encryption

  • SSE-S3 — S3-managed keys (default)
  • SSE-KMS — AWS KMS-managed keys with audit trail
  • SSE-C — Customer-provided keys
  • In Transit — TLS/HTTPS encryption

Best Practice: Always keep Block Public Access enabled unless you have a specific, justified reason to allow public access (such as static website hosting).

S3 security operates at multiple layers. Bucket policies control who can access the bucket and objects. IAM policies control what actions a principal can perform. ACLs are a legacy mechanism — AWS recommends disabling them and using policies instead. For encryption, SSE-S3 is now applied by default to all new objects. SSE-KMS gives you more control and an audit trail via CloudTrail. SSE-C means you manage the keys entirely. All data should be transferred over HTTPS.

S3 Strong Consistency Model

Since December 2020, Amazon S3 provides strong read-after-write consistency for all operations — at no additional cost.

What This Means

  • A GET immediately after a PUT returns the latest version
  • A LIST immediately reflects newly added or deleted objects
  • Applies to all S3 operations

Impact

Applications no longer need to build retry logic or wait for data to become consistent. This simplifies architectures that depend on S3 for shared state or data lakes.

Before December 2020, S3 only provided eventual consistency for overwrite PUTs and DELETEs. This meant you might read stale data briefly after an update. Now, S3 delivers strong consistency automatically. This was a major improvement that simplified data lake and analytics architectures. For the exam, remember: S3 is now strongly consistent for all operations.

S3 Use Cases

Backup and Restore

Store backups with versioning and lifecycle policies. Use cross-region replication for disaster recovery.

Static Website Hosting

Host static HTML, CSS, and JavaScript directly from an S3 bucket. Often paired with CloudFront for global delivery.

Data Lake

Central repository for structured and unstructured data at any scale. Query directly with Athena, Redshift Spectrum, or EMR.

Disaster Recovery

Cross-region replication ensures data availability even if an entire region goes down. Combined with S3 Object Lock for compliance.

S3 is incredibly versatile. Backup and restore is the most common use case. Static website hosting is popular for single-page applications and marketing sites. Data lakes on S3 form the foundation of modern analytics — services like Athena let you query data in place without loading it into a database. For disaster recovery, cross-region replication keeps a copy of your data in another region automatically.

S3 Transfer Acceleration

S3 Transfer Acceleration uses Amazon CloudFront edge locations to accelerate uploads to S3 over long distances.

How It Works

  • Client uploads to the nearest CloudFront edge location
  • Data is then transferred over the optimized AWS backbone network
  • Can improve upload speeds by 50-500% for distant transfers

When to Use

  • Users uploading from around the world
  • Transferring large objects over long distances
  • Bucket is in a distant region from uploaders
Transfer Acceleration is particularly useful when you have a centralized bucket but global users. Instead of traversing the public internet the entire way, the upload goes to the nearest edge location and then uses the fast AWS internal network. There is an additional per-GB charge. AWS provides a speed comparison tool to test if acceleration will benefit your specific use case.

Additional Storage Services

EBS, EFS, FSx, Storage Gateway & Snow Family

Now let us look beyond S3 at the other AWS storage services. These cover block storage, file storage, hybrid solutions, and physical data transfer options.

Amazon EBS Volume Types

Amazon Elastic Block Store (EBS) provides persistent block-level storage volumes for EC2 instances. Volumes are replicated within a single Availability Zone.

Volume Type Category Max IOPS Best For
gp3 General Purpose SSD 16,000 Most workloads, boot volumes
io2 Block Express Provisioned IOPS SSD 256,000 Mission-critical, large databases
st1 Throughput Optimized HDD 500 MB/s Big data, data warehouses, logs
sc1 Cold HDD 250 MB/s Infrequently accessed, lowest cost
EBS volumes are essentially virtual hard drives in the cloud. gp3 is the default and most commonly used — it provides a good balance of price and performance. io2 Block Express is for the most demanding I/O workloads like large relational databases. HDD volumes (st1 and sc1) cannot be used as boot volumes but are much cheaper for throughput-oriented or cold workloads. Remember: EBS is tied to a single AZ.

Instance Store vs EBS

Characteristic Instance Store (Ephemeral) EBS (Persistent)
Persistence Data lost on stop, terminate, or failure Persists independently of instance lifecycle
Backup No built-in backup mechanism Supports snapshots for point-in-time backups
Cost Included in instance price Separate charge based on volume type and size
Use Case Buffers, caches, scratch data, temporary content Databases, boot volumes, persistent application data

Important: Instance store data is LOST when the instance is stopped or terminated — this is a frequently tested concept.

This comparison is critical for the exam. Instance store volumes are physically attached to the host machine, giving them the best possible I/O performance. However, the data is ephemeral — if the instance stops, terminates, or the underlying hardware fails, all data on the instance store is lost. EBS volumes are network-attached and persist independently. You can stop an instance and start it again without losing EBS data. You can also detach an EBS volume and attach it to a different instance in the same Availability Zone. For anything that needs to survive an instance lifecycle event, always use EBS.

EBS Snapshots

EBS Snapshots are point-in-time backups of EBS volumes, stored in Amazon S3. They capture the state of a volume at the moment the snapshot is initiated.

Snapshot Features

  • Incremental — only changed blocks are saved after the first snapshot
  • Stored durably in Amazon S3
  • Persist independently even after the source volume is deleted
  • Automated via Amazon Data Lifecycle Manager

Snapshot Use Cases

  • Backup and restore of EBS volumes
  • Cross-AZ migration — create a volume from a snapshot in another AZ
  • Cross-region disaster recovery — copy snapshots to other regions
  • AMI creation from snapshot-backed volumes

Key Concept: Snapshots are the primary mechanism for backing up EBS volumes and migrating data across AZs and Regions.

EBS snapshots are essential for data protection and migration strategies. The first snapshot is a full copy, but subsequent snapshots are incremental — only the blocks that have changed since the last snapshot are saved. This makes them storage-efficient and cost-effective. You can create a new EBS volume from a snapshot in any Availability Zone within the same region, which is how you migrate data between AZs. You can also copy snapshots across regions for disaster recovery. Amazon Data Lifecycle Manager automates the creation, retention, and deletion of snapshots based on policies you define. Snapshots can also be shared with other AWS accounts for collaboration or multi-account architectures.

Amazon EFS

Elastic File System

Fully managed, elastic NFS file system that scales automatically. Designed for Linux workloads.

Key Features

  • Shared across multiple EC2 instances and AZs
  • Grows and shrinks automatically
  • Supports NFS v4.0 and v4.1
  • Petabyte-scale capacity

Storage Classes

  • Standard — Frequently accessed data
  • Standard-IA — Infrequent access, lower cost
  • One Zone — Single AZ, lower cost
  • One Zone-IA — Single AZ infrequent access

Note: EFS is Linux only. For Windows workloads, use Amazon FSx for Windows File Server.

EFS is a fully managed NFS file system. Unlike EBS, which attaches to a single instance, EFS can be mounted by thousands of instances across multiple Availability Zones simultaneously. It is elastic — you pay only for the storage you use, and it scales automatically. EFS supports lifecycle management to move infrequently accessed files to the IA tier. Important: EFS only supports Linux. For Windows, you need FSx.

Amazon FSx

FSx for Windows File Server

  • Fully managed Windows native file system
  • Built on Windows Server with SMB protocol
  • Integrates with Active Directory
  • Supports Windows features: DFS, ACLs, shadow copies
  • Ideal for Windows application migrations

FSx for Lustre

  • High-performance parallel file system
  • Sub-millisecond latencies
  • Hundreds of GB/s throughput
  • Seamless integration with S3
  • Ideal for HPC, ML training, media processing

FSx provides managed, purpose-built file systems — choose the flavor that matches your workload requirements.

Amazon FSx provides fully managed third-party file systems. FSx for Windows File Server is the go-to solution for Windows workloads that need shared file storage with SMB protocol and Active Directory integration. FSx for Lustre is designed for compute-intensive workloads — high-performance computing, machine learning, and video processing. Lustre can also be linked to an S3 bucket, allowing you to process S3 data with a high-performance file system interface. AWS also offers FSx for NetApp ONTAP and FSx for OpenZFS.

AWS Storage Gateway

Storage Gateway is a hybrid cloud storage service that provides on-premises applications with access to virtually unlimited cloud storage.

S3 File Gateway

Presents S3 objects as files via NFS/SMB. Local cache for low-latency access.

Volume Gateway

Block storage volumes backed by S3 snapshots. Cached or stored mode.

Tape Gateway

Virtual tape library backed by S3 and Glacier. Replaces physical tape infrastructure.

Storage Gateway bridges on-premises environments with AWS cloud storage. It runs as a virtual machine or hardware appliance in your data center. S3 File Gateway is the most common — it lets on-premises applications read and write files that are stored as S3 objects. Volume Gateway provides iSCSI block storage backed by S3. Tape Gateway replaces physical tape backup systems. This is a key hybrid storage service for the exam.

AWS Snow Family

Physical devices for large-scale data transfer when network transfer is impractical due to bandwidth, cost, or time constraints.

Snowcone

  • Small, rugged, portable
  • 8 TB HDD or 14 TB SSD
  • Edge computing capable
  • Can ship or use DataSync

Snowball Edge

  • Storage Optimized: 80 TB
  • Compute Optimized: 42 TB
  • Local compute and storage
  • Clusterable for larger jobs

Snowmobile

  • 45-foot shipping container
  • Up to 100 PB per Snowmobile
  • For exabyte-scale migrations
  • GPS tracked, 24/7 security
The Snow Family solves the problem of moving very large amounts of data to AWS when network transfer would take too long or cost too much. Snowcone is the smallest — suitable for edge locations and IoT data collection. Snowball Edge is the most commonly used for large migrations — you can cluster multiple devices together. Snowmobile is literally a truck and is for extreme cases like data center migrations at the petabyte to exabyte scale. The rule of thumb: if a transfer would take more than a week over your network, consider Snow devices.

Key Takeaways

  • Three storage types: Block (EBS), File (EFS, FSx), and Object (S3) — each optimized for different workloads
  • Amazon S3 delivers 11 nines of durability, strong consistency, and seven storage classes to optimize cost
  • S3 security is multi-layered: bucket policies, IAM policies, encryption at rest (SSE-S3/KMS/C), and encryption in transit (TLS)
  • Lifecycle policies and Intelligent-Tiering are the primary tools for automating S3 cost optimization
  • EBS is single-AZ block storage; EFS is multi-AZ shared file storage for Linux; FSx serves Windows and HPC needs
  • Storage Gateway bridges on-premises to cloud; Snow Family handles large-scale physical data transfer
