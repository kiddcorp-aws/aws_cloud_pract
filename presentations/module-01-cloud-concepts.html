What is Cloud Computing

Foundations and Definitions

We begin with the fundamentals of cloud computing. Understanding these concepts is essential before diving into AWS-specific services.

Overview of Cloud Computing

Traditional IT (CapEx)

  • Purchase physical hardware upfront
  • Maintain data centers and staff
  • Capacity planning and over-provisioning
  • Long procurement cycles (weeks to months)

Cloud Computing (OpEx)

  • On-demand delivery of IT resources
  • Pay only for what you consume
  • Scale up or down instantly
  • Provision in minutes, not months
Key Shift: Cloud computing transforms IT from a capital expense (CapEx) to an operational expense (OpEx), enabling organizations to focus on innovation rather than infrastructure management.
Emphasize the fundamental shift from CapEx to OpEx. In the traditional model, organizations had to make large upfront investments in hardware, estimate capacity years in advance, and maintain costly data center facilities. Cloud computing eliminates this by letting you pay for resources as you use them, much like a utility bill.

NIST Cloud Definition

The National Institute of Standards and Technology defines five essential characteristics:

  • On-Demand Self-Service -- Provision resources automatically without human interaction with the provider
  • Broad Network Access -- Resources available over the network via standard mechanisms
  • Resource Pooling -- Provider resources pooled across multiple consumers (multi-tenant)
  • Rapid Elasticity -- Scale resources up and down with demand, appearing unlimited
  • Measured Service -- Automatic metering and optimization of resource usage
The NIST definition (SP 800-145) is widely accepted as the standard definition of cloud computing. These five characteristics are what distinguish true cloud services from traditional hosting. Note that all five must be present for a service to be considered cloud computing.

Cloud Service Models

Aspect IaaS PaaS SaaS
What You Manage OS, middleware, runtime, apps, data Apps and data Nothing (just use it)
Provider Manages Virtualization, servers, storage, networking OS, middleware, runtime, plus infra Everything
AWS Examples EC2, EBS, VPC Elastic Beanstalk, RDS, Lambda WorkMail, Chime, QuickSight
Use Case Full control over infrastructure Focus on application development End-user applications
Flexibility Highest Moderate Lowest
Key Point: As you move from IaaS to SaaS, you manage less and the provider manages more. The trade-off is flexibility versus ease of use.
Walk through the table row by row. IaaS gives you the most control but the most responsibility. PaaS lets you focus on your code while the provider manages the platform. SaaS is a complete application delivered over the internet. Most organizations use a mix of all three models.

Cloud Deployment Models

Public Cloud

Resources owned and operated by a third-party provider, delivered over the internet. Examples: AWS, Azure, GCP.

Private Cloud

Cloud infrastructure operated solely for a single organization. Higher cost but greater control and compliance.

Hybrid Cloud

Combines public and private clouds with data and application portability. Most enterprises adopt this model.

Community Cloud

Shared infrastructure for a specific community with common concerns such as security and compliance.

Public cloud is what most people think of when they hear "cloud." Private cloud is often used by organizations with strict regulatory requirements. Hybrid cloud is increasingly popular as organizations migrate gradually. AWS supports all of these models -- for example, AWS Outposts brings AWS infrastructure on-premises for hybrid deployments.

Introduction to AWS

The Market Leader in Cloud Services

Now that we understand cloud computing fundamentals, let us look at AWS specifically -- the largest and most mature cloud provider in the world.

Amazon Web Services Overview

AWS launched in 2006 and has grown to become the world's most comprehensive and broadly adopted cloud platform, serving millions of customers worldwide.

200+ Services

Compute, storage, databases, machine learning, IoT, security, and more.

30+ Regions

Global presence across the Americas, Europe, Asia Pacific, Middle East, and Africa as of 2026.

600+ Edge Locations

Content delivery and edge computing locations worldwide for low-latency access.

Market Position: AWS holds the largest share of the global cloud infrastructure market, ahead of Microsoft Azure and Google Cloud Platform.
AWS started with simple services like S3 (storage) and EC2 (compute) and has since expanded to over 200 fully featured services. The breadth and depth of AWS services is unmatched. As of 2026, AWS has more than 30 Regions globally, each containing multiple Availability Zones.

Multi-Tenancy and Shared Infrastructure

What is Multi-Tenancy?

Multiple customers share the same physical infrastructure while maintaining logical isolation of their resources. Each customer's data and configurations are invisible to others.

Benefits

  • Lower costs through economies of scale
  • Provider handles maintenance and upgrades
  • Better resource utilization
  • Rapid provisioning of new resources

How AWS Ensures Isolation

  • Hypervisor-level isolation for compute
  • Network isolation via VPC
  • Encryption at rest and in transit
  • IAM policies for access control
Multi-tenancy is what makes cloud computing economically viable. By sharing physical infrastructure across many customers, AWS achieves economies of scale that individual organizations cannot. The key concern is security -- AWS uses the Nitro System hypervisor, network segmentation, and encryption to ensure strict isolation between tenants.

AWS Value Propositions

On-Demand Provisioning

Launch resources in minutes with no long-term commitments.

Elasticity

Scale resources automatically with demand.

Pay-As-You-Go

Pay only for what you consume, with no upfront costs.

Global Reach

Deploy in multiple regions worldwide to reduce latency.

Innovation

Access AI/ML, IoT, and serverless without building from scratch.

Security

Secure foundation with compliance certifications and extensive controls.

These are the core reasons organizations choose AWS. On-demand provisioning eliminates the wait time of traditional IT. Elasticity means you do not have to guess capacity. Pay-as-you-go aligns costs with actual usage. Global reach lets you serve customers anywhere. AWS continuously innovates, releasing thousands of new features annually.

Six Advantages of Cloud Computing

1. Trade Fixed Expense for Variable Expense

Instead of investing heavily in data centers before knowing how you will use them, pay only when you consume computing resources.

2. Benefit from Massive Economies of Scale

AWS aggregates usage from hundreds of thousands of customers, achieving higher economies of scale and lower pay-as-you-go prices.

3. Stop Guessing Capacity

Eliminate guessing on infrastructure capacity needs. Scale up or down as required with only a few minutes notice.

4. Increase Speed and Agility

New IT resources are a click away, reducing the time to make those resources available from weeks to minutes.

5. Stop Spending Money Running Data Centers

Focus on projects that differentiate your business rather than on the undifferentiated heavy lifting of racking, stacking, and powering servers.

6. Go Global in Minutes

Deploy your application in multiple AWS Regions around the world with just a few clicks, providing lower latency and a better experience for customers.

Exam Tip: These six advantages are directly from AWS and frequently tested on the CLF-C02 exam.
These six advantages come directly from AWS whitepapers and are a staple of the Cloud Practitioner exam. Students should be able to recall and explain each one. Emphasize the shift from CapEx to OpEx (advantage 1), the benefit of AWS pooling demand across customers (advantage 2), and how elasticity removes the need to guess capacity (advantage 3). Speed, agility, and global reach are what make cloud transformative for organizations of all sizes.

AWS Global Infrastructure

Regions, Availability Zones, and Edge Locations

The AWS global infrastructure is the foundation upon which all AWS services are built. Understanding this infrastructure is critical for designing resilient, high-performing architectures.

Regions, Availability Zones, and Edge Locations

Regions

  • Geographic areas with multiple data centers
  • 30+ Regions worldwide (as of 2026)
  • Fully independent and isolated
  • Data does not replicate between Regions unless you configure it

Availability Zones (AZs)

  • One or more discrete data centers
  • Each Region has 2-6 AZs
  • Connected via low-latency links
  • Physically separated for fault tolerance

Edge Locations

  • 600+ points of presence globally
  • Used by CloudFront (CDN) and Route 53 (DNS)
  • Cache content close to users
  • Reduce latency for end users
Regions are the largest unit of AWS infrastructure. Each Region is completely independent, which means a failure in one Region does not affect others. Within each Region, Availability Zones are physically separated data centers with independent power, cooling, and networking. Edge locations are used for content delivery and are far more numerous than Regions.

The AWS Distributed Architecture

Built for Resilience

AWS infrastructure is designed so that no single point of failure can take down an application. By distributing workloads across multiple Availability Zones and Regions, you achieve high availability and fault tolerance.

High Availability

  • Deploy across multiple AZs within a Region
  • Automatic failover with services like ELB and RDS Multi-AZ
  • SLA guarantees of 99.99% for many services

Disaster Recovery

  • Cross-Region replication for critical data
  • Multi-Region active-active architectures
  • AWS Backup for centralized backup management
The distributed architecture is one of the biggest advantages of AWS. In traditional data centers, achieving this level of redundancy requires enormous capital investment. With AWS, you can architect for high availability by simply deploying across multiple AZs. For disaster recovery, you can replicate data to another Region.

How to Choose a Region

Consider these four factors when selecting an AWS Region:

1. Compliance

Does your data need to stay within a specific geographic boundary? Regulations like GDPR may dictate where data must reside.

This is often the most important factor and should be evaluated first.

2. Latency

Choose a Region close to your end users to minimize network latency and improve user experience.

3. Service Availability

Not all AWS services are available in every Region. Verify that the services you need are offered in your chosen Region.

4. Pricing

Pricing varies by Region. Some Regions are more expensive due to local costs. Compare pricing for your primary services.

Region selection is a foundational architectural decision. Compliance always comes first -- if regulations require data to stay in a country, that narrows your options immediately. After compliance, proximity to users reduces latency. Check that needed services exist in the Region. Finally, pricing can vary significantly -- for example, the Sao Paulo Region is typically more expensive than US East (N. Virginia).

AWS Cloud Adoption Framework (CAF)

The CAF organizes guidance into six areas of focus, called perspectives:

Business

Aligns IT investment with business outcomes. Ensures IT strategy drives business results.

People

Addresses HR, staffing, and organizational change management needed for cloud adoption.

Governance

Aligns IT strategy with business strategy. Maximizes the value of IT while minimizing risk.

Platform

Covers architecture and infrastructure design. Helps build an enterprise-grade, scalable cloud platform.

Security

Ensures the organization meets security objectives for visibility, auditability, control, and agility.

Operations

Defines how day-to-day, quarter-to-quarter, and year-to-year business operations are conducted in the cloud.

Why CAF Matters

The Cloud Adoption Framework helps organizations identify gaps in skills and processes during cloud adoption. It provides a structured approach to planning, ensuring that business, people, and technology perspectives are all addressed before and during migration.

The AWS Cloud Adoption Framework is a guidance tool, not a service. It helps organizations understand how adopting the cloud will affect their business, people, and technology. The six perspectives split into two groups: Business, People, and Governance are business-oriented; Platform, Security, and Operations are technical. Each perspective identifies gaps in skills and processes that need to be addressed. This is tested on the CLF-C02 exam, so students should know the six perspective names and their general purpose.

Migration Strategies -- The 7 R's

Strategy Description Example
Rehost Lift and shift -- move applications as-is to the cloud Migrating on-premises VMs directly to Amazon EC2
Replatform Lift, tinker, and shift -- make minor optimizations during migration Moving a database to Amazon RDS instead of self-managing on EC2
Refactor / Re-architect Redesign the application using cloud-native features Breaking a monolith into microservices using Lambda and containers
Repurchase Move to a different product, typically a SaaS solution Replacing an on-premises CRM with Salesforce
Retire Decommission applications that are no longer needed Shutting down legacy systems with no active users
Retain Keep in the current environment for now; revisit later Mainframe applications with complex dependencies
Relocate Move infrastructure to AWS without changes at the hypervisor level Migrating VMware workloads to VMware Cloud on AWS
Key Insight: Most large migrations start with Rehost for quick wins, then optimize with Replatform/Refactor over time.
The 7 R's provide a framework for deciding how to handle each application during a migration. Rehost is the fastest path and works well for organizations that need to vacate a data center quickly. Replatform makes targeted optimizations without a full re-architecture. Refactor delivers the most cloud-native benefits but requires the most effort. Repurchase is common when a SaaS product replaces custom-built software. Retire and Retain reduce the scope of the migration. Relocate was added more recently and applies specifically to VMware workloads. On the exam, students should recognize each strategy and match it to the appropriate scenario.

Shared Responsibility Model

Security OF the Cloud vs. Security IN the Cloud

The Shared Responsibility Model is one of the most important concepts for the Cloud Practitioner exam. It defines what AWS is responsible for and what the customer is responsible for.

Shared Responsibility Model

AWS Responsibility: Security OF the Cloud

  • Physical security of data centers
  • Hardware and global infrastructure
  • Managed services software patching
  • Networking infrastructure
  • Hypervisor and virtualization layer
  • Compliance validation of infrastructure
AWS secures the foundation upon which your services run.

Customer Responsibility: Security IN the Cloud

  • Customer data and encryption
  • Identity and access management (IAM)
  • Operating system and firewall configuration
  • Network traffic protection and encryption
  • Application-level security
  • Security group and NACL configuration
You are responsible for properly configuring and securing your workloads.
The simplest way to remember this: AWS is responsible for the security OF the cloud (the infrastructure), while the customer is responsible for security IN the cloud (their data, configurations, and applications). The exact division depends on the service model -- with IaaS like EC2, the customer has more responsibility. With managed services like RDS or Lambda, AWS takes on more.

Responsibility Varies by Service Type

Responsibility IaaS (EC2) PaaS (RDS) Serverless (Lambda)
Physical Infrastructure AWS AWS AWS
Operating System Customer AWS AWS
Patching Customer AWS AWS
Network Configuration Customer Shared AWS
Application Code Customer Customer Customer
Data Encryption Customer Customer Customer
IAM / Access Control Customer Customer Customer
Key Point: The more managed the service, the more AWS handles. However, the customer is always responsible for their data, access control, and application-level security -- regardless of the service model.
This table illustrates how responsibility shifts depending on the service type. With EC2, you manage the OS, patching, and network configuration. With RDS, AWS handles the database engine patching and OS. With Lambda, AWS manages virtually everything except your code and data. But notice that data encryption and IAM are always the customer's responsibility.

AWS Services Overview

Categories, Managed Services, and Access Methods

Let us take a broad look at the major AWS service categories and how you interact with them. We will explore individual services in detail throughout the rest of the course.

Managed vs. Unmanaged Services

Unmanaged Services

You are responsible for scaling, availability, and fault tolerance.

  • Amazon EC2 -- you manage the instance, OS, and applications
  • Greater control and flexibility
  • More operational overhead
You handle provisioning, patching, scaling, and backups.

Managed Services

AWS handles scaling, availability, and maintenance automatically.

  • Amazon RDS, DynamoDB, Lambda, S3
  • Less operational overhead
  • Built-in high availability and durability
AWS handles the undifferentiated heavy lifting for you.
Understanding the distinction between managed and unmanaged services is important because it directly affects your operational burden. With unmanaged services like EC2, you have full control but must handle everything from OS patching to scaling. Managed services like RDS or Lambda let AWS handle infrastructure concerns so you can focus on your application.

Accessing AWS

There are three primary ways to interact with AWS services:

AWS Management Console

  • Web-based graphical interface
  • Point-and-click service management
  • Best for exploration and learning
  • Mobile app available
Ideal for getting started and visual management.

AWS CLI

  • Command-line interface
  • Scriptable and automatable
  • Available for Windows, macOS, Linux
  • Supports all AWS services
Best for scripting, automation, and power users.

AWS SDKs

  • Language-specific APIs
  • Python, Java, JavaScript, Go, .NET, and more
  • Embed AWS calls in your applications
  • Handles auth, retries, and serialization
Best for integrating AWS into applications.
All three methods use the same underlying AWS APIs, so the results are identical regardless of how you access a service. The Console is great for learning and ad-hoc tasks. The CLI is essential for automation and scripting. SDKs are used when building applications that interact with AWS services programmatically. Key Takeaways

  • Cloud computing delivers IT resources on demand, shifting from CapEx to OpEx, and is defined by five NIST characteristics.
  • Service models (IaaS, PaaS, SaaS) determine the division of management responsibility between provider and customer.
  • AWS is the market-leading cloud provider with 200+ services, 30+ Regions, and a global edge network.
  • AWS Global Infrastructure is built on Regions, Availability Zones, and Edge Locations for resilience and performance.
  • The Shared Responsibility Model defines that AWS secures the cloud infrastructure while customers secure their workloads within it.
  • AWS services range from unmanaged (EC2) to fully managed (Lambda, S3), and can be accessed via Console, CLI, or SDKs.
